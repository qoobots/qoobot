## 1. 风险管理方法论

### 1.1 管理原则

```
风险管理的核心不是"消除风险"，而是"让风险可见、可控、可应对"。
```

| 原则 | 说明 |
|------|------|
| **提前暴露** | 任何人在任何时间都可以标记风险，不设门槛 |
| **量化评估** | 每个风险必须有概率(P)和影响(I)两个维度 |
| **分级管理** | 按严重程度分 P0/P1/P2 三档，匹配不同响应级别 |
| **持续跟踪** | 风险不是一次性的，需 Sprint 级持续审查 |
| **预案先行** | 每个高危风险必须有可执行的缓解预案 |

### 1.2 风险分级标准

| 级别 | 触发条件 | 响应要求 | 升级路径 |
|------|----------|----------|----------|
| **🔴 P0** | P≥中 且 I≥高，或 P≥高 且 I≥中 | 立即制定缓解计划，每周跟踪 | 上报 Tech Lead + PM |
| **🟡 P1** | P≥中 且 I=中，或 P=低 且 I=高 | Sprint 内制定计划，双周跟踪 | Tech Lead 关注 |
| **🟢 P2** | P=低 且 I=中，或 P=中 且 I=低 | 记录在案，按需关注 | 模块负责人关注 |

### 1.3 概率与影响量级

| 概率 | 含义 | 量化参考 |
|------|------|----------|
| 高 (H) | 大概率发生 | > 50% |
| 中 (M) | 有可能发生 | 20% - 50% |
| 低 (L) | 不太可能发生 | < 20% |

| 影响 | 含义 | 量化参考 |
|------|------|----------|
| 高 (H) | 严重影响里程碑 | 延期 ≥ 2 个月 或 核心功能无法交付 |
| 中 (M) | 局部影响 | 延期 2-4 周 或 单项功能降级 |
| 低 (L) | 轻微影响 | 延期 ≤ 1 周 或 体验层面影响 |

## 3. 技术风险深度分析

### R01：LLM 推理延迟过高

| 属性 | 内容 |
|------|------|
| **编号** | R01 |
| **级别** | 🔴 P0 |
| **概率** | 中 (30%) |
| **影响** | 高 — 认知引擎成为全链路瓶颈，用户体验不可接受 |

**场景描述**：
Qwen2.5-7B 在 Jetson Orin AGX 上的 INT4 推理目标延迟 < 500ms。如果实际延迟超出目标（如 > 2s），用户从发出指令到看到反馈的等待时间过长，不符合交互预期。

**触发条件**：
- Jetson Orin 实际可用显存 < 预期（被 ROS 2 / SLAM 占用后不足）
- TensorRT-LLM 编译 Qwen2.5-7B 失败或性能不达预期
- 仅能评估 20 tokens 对应 ~1s，复杂任务需要 100+ tokens
- 量化后模型质量下降，需回退 FP16（延迟翻倍）

**缓解措施**：

| 层级 | 措施 | 负责人 | 时间 |
|------|------|--------|------|
| 预防 | Sprint 2 Week 1 即启动 7B/3B/1.5B 三尺寸 Benchmark（见 ADR-004） | AI 工程师 | Sprint 2 W1 |
| 预防 | 设计 `ModelLoader` 抽象接口，支持运行时热切换模型尺寸 | AI 工程师 | Sprint 2 W2 |
| 降级 | 自动降级：延迟 > 500ms → 切换 Qwen2.5-3B → 仍超时 → 切换 1.5B | AI 工程师 | Sprint 2 |
| 兜底 | CPU 模式：llama.cpp 运行 1.5B，保证可用性（牺牲质量） | AI 工程师 | Sprint 2 W3 |
| 兜底 | 云端降级：本地超时 5s → 静默切换到 DeepSeek-V3 API | AI 工程师 | Sprint 2 |

**监控指标**：

```
Metric: llm_inference_latency_p50 / p95 / p99
Target: p50 < 300ms, p95 < 800ms, p99 < 1.5s
Alert:  p50 > 500ms for 5min → 触发降级
```

### R02：仿真与真实差异（Sim-to-Real Gap）

| 属性 | 内容 |
|------|------|
| **编号** | R02 |
| **级别** | 🔴 P0 |
| **概率** | 高 (60%) |
| **影响** | 高 — 仿真验证的策略在真实硬件上完全不可用 |

**场景描述**：
Phase 1 全部在 Gazebo 仿真中开发和验证。传感器噪声、物理动力学、光照条件等与真实环境有显著差异。Phase 1 末期 Demo 如果在真实机械臂上运行失败，将严重影响项目信心和后续融资。

**触发条件**：
- Gazebo 物理引擎精度不足（碰撞、摩擦力模型简化）
- 仿真光照/纹理过于理想化（YOLO 检测率虚高）
- 传感器模型无噪声（SLAM 定位精度虚高）
- 抓取成功率在仿真中 > 90%，真实 < 50%

**缓解措施**：

| 层级 | 措施 | 负责人 | 时间 |
|------|------|--------|------|
| 预防 | 仿真中注入高斯噪声 + 随机光照 + 随机障碍物（域随机化） | 仿真实工程师 | Sprint 3 |
| 预防 | 使用 Gazebo 的高保真传感器插件（`libgazebo_ros_camera` with noise model） | C++ 工程师 | Sprint 3 |
| 降级 | 仅验证高层逻辑（任务分解、行为树生成），运动学层面不做真实硬件验证 | Tech Lead | Phase 1 |
| 降级 | Phase 1 Demo 强调 "仿真环境中的端到端"，管理预期 | PM | Sprint 6 |
| 兜底 | Phase 2 首批任务：在真实机械臂上验证 Phase 1 核心链路 | 全员 | Phase 2 W1-W4 |

**指标**：

```
Metric: sim2real_transfer_success_rate (Phase 2 实测)
Target: 核心链路（感知→规划→执行）真机成功率 > 60%
Alert:  < 40% → 启动专项攻关
```

### R03：VLA 模型效果不达标

| 属性 | 内容 |
|------|------|
| **编号** | R03 |
| **级别** | 🔴 P0 |
| **概率** | 中 (35%) |
| **影响** | 高 — 核心卖点"端到端视觉-语言-动作"无法演示 |

**场景描述**：
自研 Brain-VLA 基于 OpenVLA-7B 微调，目标是中文指令直接输出动作序列。OpenVLA 本身仍在快速迭代（2024 年才发布），中文微调数据稀缺。如果 Brain-VLA 输出动作质量差或完全不可用，Brain OS 最核心的差异化能力将受损。

**触发条件**：
- OpenVLA 在中文指令微调后效果退化
- 跨本体迁移（机械臂→人形）效果差
- 推理速度不满足实时要求
- 动作序列抖动/不安全

**缓解措施**：

| 层级 | 措施 | 负责人 | 时间 |
|------|------|--------|------|
| 预防 | Phase 1 不依赖 VLA —— 核心链路使用 LLM + 规则 + 传统规划（安全路径） | Tech Lead | Phase 1 全程 |
| 预防 | 建立中文机器人指令数据集（先人工标注 1000 条） | AI 工程师 | Phase 2 |
| 降级 | VLA 降级为"建议模式"——输出作为参考，不直接执行 | AI 工程师 | Phase 2 |
| 兜底 | 保持 LLM + MoveIt 2 经典管路作为绝对兜底 | C++/AI 工程师 | 持续 |
| 兜底 | 人机协作模式：VLA 不可用时人工操作（HITL） | 前端工程师 | Sprint 5 |

### R04：多模态感知融合失败

| 属性 | 内容 |
|------|------|
| **编号** | R04 |
| **级别** | 🔴 P0 |
| **概率** | 中 (30%) |
| **影响** | 高 — SLAM + 3DGS + YOLO + 位姿估计各管各的，无法形成统一场景理解 |

**场景描述**：
感知管线涉及 4 个独立模型（ORB-SLAM3、3DGS、YOLOv11、FoundationPose），各自输出不同格式、不同坐标系的感知结果。如果融合层无法将它们统一为一致的场景表示，决策层将收到矛盾信息。

**触发条件**：
- SLAM 和 3DGS 坐标系原点不一致
- YOLO 检测框与 3DGS 点云不对齐
- 多传感器时间戳不同步
- 感知融合模块过度设计、开发周期过长

**缓解措施**：

| 层级 | 措施 | 负责人 | 时间 |
|------|------|--------|------|
| 预防 | Phase 1 简化路径：SLAM 定位 + YOLO 检测 → 2D 场景图（不融合 3DGS） | C++ 工程师 | Sprint 3 |
| 预防 | 统一时间戳源（ROS 2 `sim_time` / 硬件时钟），传感器间强制同步 | C++ 工程师 | Sprint 3 |
| 降级 | Phase 1 不使用 FoundationPose，用物体位置近似 + 固定抓取姿态 | C++ 工程师 | Sprint 3 |
| 降级 | 3DGS 降级为传统点云（Point Cloud），Phase 2 再升级 | C++ 工程师 | Sprint 5 |
| 兜底 | 如果融合失败 → 仅使用 YOLO 2D 目标位置（Z 轴固定假设） | Tech Lead | anytime |

### R05：实时通信链路断裂

| 属性 | 内容 |
|------|------|
| **编号** | R05 |
| **级别** | 🔴 P0 |
| **概率** | 中 (25%) |
| **影响** | 高 — 感知数据无法到达决策层，或控制指令无法下发，系统不可用 |

**场景描述**：
Brain OS 依赖 C++ ↔ Python (gRPC) 和 Python ↔ Web (WebSocket) 两条跨进程通信链。在高频数据（30Hz 图像 / 100Hz 关节状态）冲击下，可能出现消息丢失、连接断开、序列化开销爆炸。

**触发条件**：
- Protobuf 消息过大（> 1MB）导致序列化延迟
- WebSocket 连接在大数据量下断开
- gRPC 流式传输在 ROS 2 高负载下超时
- ROS 2 DDS 发现协议在重连后不稳定

**缓解措施**：

| 层级 | 措施 | 负责人 | 时间 |
|------|------|------|------|
| 预防 | Phase 1 仅传输结构化数据（意图/行为树/轨迹），不传输原始图像/点云 | C++ 工程师 | Sprint 1 |
| 预防 | gRPC 消息体限制：单次 < 64KB，超限走分块传输 | C++/AI 工程师 | Sprint 1 |
| 预防 | WebSocket 心跳 + 自动重连（前端 3 次重试 + exponential backoff） | 前端工程师 | Sprint 5 |
| 降级 | 通信链路中断 > 500ms → 触发安全引擎急停 | C++ 工程师 | Sprint 3 |
| 降级 | gRPC 不可用时 WebSocket 承载全部跨语言通信 | AI 工程师 | Sprint 5 |
| 兜底 | 全数据走 ROS 2 Topic（同机 DDS 共享内存，延迟 < 10μs） | C++ 工程师 | 持续 |

### R07：3DGS 实时重建性能不足

| 属性 | 内容 |
|------|------|
| **编号** | R07 |
| **级别** | 🔴 P0 |
| **概率** | 中 (40%) |
| **影响** | 高 — 前端无法获得实时 3D 场景底图，可视化效果大幅降级 |

**场景描述**：
3D Gaussian Splatting 训练和渲染对 GPU 要求高。Jetson Orin AGX 的 GPU 性能约为桌面 RTX 4090 的 1/8。实时训练（online training）几乎不可能，离线预训练 + 增量更新也可能达不到 10fps。

**触发条件**：
- 离线训练时间 > 1 分钟（不适合动态环境）
- 增量更新帧率 < 5fps
- 3DGS 点云过大（> 100 万 Gaussian），前端 gsplat.js 渲染掉帧

**缓解措施**：

| 层级 | 措施 | 负责人 | 时间 |
|------|------|--------|------|
| 预防 | Phase 1 不追求实时重建——使用预训练 3DGS 场景（静态环境） | AI 工程师 | Sprint 3 |
| 降级 A | 降级为传统点云：Point Cloud → ROS 2 Topic → Three.js PointsMaterial | C++ 工程师 | Sprint 5 |
| 降级 B | 仅增量更新变动区域（前景物体），背景 3DGS 不更新 | AI 工程师 | Phase 2 |
| 降级 C | GPU 渲染 → H.264 视频流 → 前端播放（放弃交互式 3D，保留实时画面） | 前端/AI 工程师 | Phase 2 |
| 兜底 | 静态 CAD 模型 + YOLO 标记（无 3D 重建，纯几何场景） | C++ 工程师 | Sprint 5 |

### R10：碰撞检测漏检

| 属性 | 内容 |
|------|------|
| **编号** | R10 |
| **级别** | 🔴 P0 |
| **概率** | 中 (25%) |
| **影响** | 高 — 机器人碰撞自身/环境/人，造成安全事故 |

**场景描述**：
FCL (Flexible Collision Library) 检测后置信区间不够，尤其在动态环境、部分遮挡、非刚体物体（布料、线缆）场景下。漏检可能导致机器人执行危险轨迹。

**缓解措施**：

| 层级 | 措施 |
|------|------|
| 预防 | 多重检测链路：FCL 碰撞检测 + 关节限位 + 力矩监控 + 速度限制（四层防护） |
| 预防 | 安全边界：在真实机器人周围设置虚拟安全区域（Virtual Fence） |
| 降级 | 任何一层触发 → 减速至安全速度（< 0.1 m/s） |
| 兜底 | 急停：力矩异常 > 阈值 或 手动物理急停按钮 → 制动 |
| 兜底 | Phase 1 仅仿真验证，无真实硬件碰撞风险 |

> 注：R10 与 SafetyMonitor 模块深度绑定，详见 03 §4.1 安全引擎设计。

### R13：模型推理显存溢出（OOM）

| 属性 | 内容 |
|------|------|
| **编号** | R13 |
| **级别** | 🔴 P0 |
| **概率** | 中 (35%) |
| **影响** | 高 — 多个 AI 模型同时加载导致显存耗尽，服务崩溃 |

**场景描述**：
Jetson Orin AGX 64GB 统一内存（CPU + GPU 共享），实际可用约 58GB。同时运行 Qwen2.5-7B (~5GB INT4)、YOLOv11 (~1GB)、SLAM (~2GB)、3DGS (~8GB)、ROS 2 (~4GB)，总和可能超出。如果未做显存管理和优先级调度，会出现 OOM 崩溃。

**缓解措施**：

| 层级 | 措施 |
|------|------|
| 预防 | 建立显存预算表：为每个模型分配最大显存配额 |
| 预防 | 模型按需加载/卸载（LLM 推理完成后释放→加载 3DGS 训练） |
| 降级 | OOM 前 80% 预警 → 卸载非关键模型（3DGS、FoundationPose） |
| 降级 | LLM 切换 3B → 释放 2GB，或切换 1.5B CPU 推理 |
| 兜底 | watchdog 检测 OOM → 自动重启 brain_ai 容器 |

**显存预算参考**：

| 组件 | 预期占用 | 优先级 | 可降级 |
|------|----------|--------|--------|
| ROS 2 + DDS | ~4 GB | 系统级 | 否 |
| Qwen2.5-7B INT4 | ~5 GB | 高 | 可降 3B (1.5GB) |
| YOLOv11 nano | ~1 GB | 高 | 否 |
| ORB-SLAM3 | ~2 GB | 高 | 否 |
| 3DGS 推理 | ~8 GB | 中 | 可降 Point Cloud |
| 3DGS 训练 | ~12 GB | 低 | 可降离线预训练 |
| **总计** | **~32 GB** | - | 安全余量 ~26GB |

## 7. 应急预案与降级链路

### 7.1 系统降级拓扑

```
正常运行
    │
    ├── 认知引擎降级 ─────────────────────────────────────┐
    │   Qwen-7B → Qwen-3B → Qwen-1.5B → DeepSeek API     │
    │                                                      │
    ├── 感知降级 ─────────────────────────────────────────┤
    │   3DGS → Point Cloud → Static CAD                    │
    │   YOLO + SAM → YOLO only → Predefined Objects       │
    │                                                      │
    ├── 决策降级 ─────────────────────────────────────────┤
    │   VLA → LLM+BT → Rule-based → Manual (HITL)         │
    │   5 轨迹 → 2 轨迹 → 1 轨迹 (最优)                    │
    │                                                      │
    ├── 通信降级 ─────────────────────────────────────────┤
    │   gRPC → WebSocket → ROS 2 DDS only                 │
    │                                                      │
    ├── 可视化降级 ───────────────────────────────────────┤
    │   3D WebGL → 2D Canvas → Text Status                │
    │                                                      │
    └── 安全降级 ─────────────────────────────────────────┤
       减速 → 限速 → 停止 → 急停 (Emergency Stop)
```

### 7.2 应急预案触发条件

| 场景 | 触发条件 | 响应动作 | RTO |
|------|----------|----------|-----|
| LLM 调用超时 | p95 > 2s for 10min | 自动切换下一级模型 | < 30s |
| gRPC 连接断开 | 心跳丢失 > 3s | 切换 WebSocket 透传 | < 5s |
| GPU OOM | 显存使用 > 85% | 卸载低优模型 | < 10s |
| 碰撞检测触发 | FCL 返回碰撞 | 减速至 20% | 实时 |
| 前端 WebSocket 断开 | ws.onclose | 自动重连（3 次）+ 显示"连接断开" | < 5s |
| 全链路故障 | 核心服务全部无响应 | 紧急停止 + 通知人工介入 | < 1s |

### 7.3 业务连续性计划（BCP）

```
最坏情况：Phase 1 Demo 前 1 周核心链路仍不通

备选方案 A（录播方案）：
  ├── 预录制仿真场景操作视频
  ├── Web 页面播放 + 同步解说
  └── 降低 Live Demo 风险

备选方案 B（缩小范围）：
  ├── 砍掉 VLA 演示，仅展示 LLM → BT → HITL
  ├── 砍掉实时 3DGS，用静态场景
  └── 保留 3 个最高光功能

备选方案 C（推迟 2 周）：
  └── 额外 Sprint 7 缓冲（预留但尽量不使用）
```

## 8. 风险监控与审查机制

### 8.1 审查节奏

| 会议 | 频率 | 内容 | 参与人 |
|------|------|------|--------|
| **Daily Standup** | 每天 | 阻塞项汇报 → 如有新风险立即标记 | 全员 |

| **风险专项评审** | 每 Phase 结束时 | 全面评估下个 Phase 风险、更新概率和影响 | Tech Lead + PM |
| **架构风险评审** | 架构变更时 | 评估新架构引入的风险 | Tech Lead + 架构师 |

### 8.2 风险生命周期

```
发现 ──→ 登记 ──→ 评估 ──→ 制定缓解计划 ──→ 执行跟踪 ──→ 关闭
  │                                              │
  │                                              ├── 有效 → 关闭
  │                                              │
  │                                              └── 无效 → 重新评估
  │                                                          │
  └──────────────────────────────────────────────────────────┘
```

### 8.3 风险看板

建议在项目管理工具（如 GitLab Issue Board / Linear / Notion）中维护风险看板：

| 列 | 说明 |
|------|------|
| **Open** | 新登记、待评估 |
| **Active** | 正在监控、缓解措施执行中 |
| **Escalated** | 升级到管理层决策 |
| **Mitigated** | 缓解措施到位、概率或影响降低 |
| **Closed** | 已消除或已发生并解决 |
| **Accepted** | 接受该风险（不做缓解） |
