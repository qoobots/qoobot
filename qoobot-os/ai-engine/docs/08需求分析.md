## 1. 需求概述

qoocore 是 QooBot 端侧推理的加速核心。机器人推理不同于云端 LLM——它需要多模型并发、亚毫秒级延迟、极端内存受限、多异构芯片协同。qoocore 是一层薄而高效的"编译-调度-执行"中间件。

## 2. 功能性需求 (FR)

### FR-100: 编译工具链

| 需求编号 | 功能 | 描述 | 优先级 |
|:----------|:-----|:-----|:-------|
| FR-101 | 模型导入 | ONNX Runtime 图解析 → IrNode 列表 + JSON 序列化 | P0 |
| FR-102 | 设备 IR | IrGraph DAG 表示 + IrBuilder Builder 模式(17种算子) + IrValidator(4项检查) | P0 |
| FR-103 | 图优化 | O1常量折叠/死代码消除 + O2算子融合(Conv+BN+ReLU) + O3内存复用分析 | P0 |
| FR-104 | 量化编译 | INT8/INT4/FP16 混合精度，PTQ校准统计 + scale/zero_point计算 | P0 |
| FR-105 | 模型剪枝 | 结构化L1/L2通道剪枝 + 非结构化幅值剪枝 + 蒸馏集成 | P1 |
| FR-106 | 离线编译 | 5步compile流程 → qoomodel打包 (128字节Header) | P0 |
| FR-107 | 编译器后端 | NPU指令生成(QNN/BPU/RKNN) + GPU kernel(CUDA/OpenCL/Vulkan) + CPU向量化(Neon/AVX512) | P0 |

### FR-200: 推理运行时

| 需求编号 | 功能 | 描述 | 优先级 |
|:----------|:-----|:-----|:-------|
| FR-201 | 统一推理引擎 | load_model→infer→infer_async→batch→multi_model 完整API | P0 |
| FR-202 | NPU 抽象层(HAL) | NpuHalLoader(dlopen) + MockNpuHal + HardwareProber(4家NPU+3家GPU) + IonAllocator | P0 |
| FR-203 | GPU 推理后端 | CUDA/OpenCL/Vulkan 三后端统一抽象 + 内核缓存 + 异步推理 | P1 |
| FR-204 | DSP 推理后端 | 高通 Hexagon / CEVA DSP 推理卸载 | P2 |
| FR-205 | CPU 推理后端 | ARM Neon / x86 SSE4.2/AVX2/AVX512 SIMD + CPUID探测 + 多线程 | P1 |
| FR-206 | 实时推理调度 | 固定优先级抢占 + 时间片轮转 + 多级队列 + 流水线并行 | P0 |
| FR-207 | 流水线执行 | 感知→认知→规划三阶段Overlap + 前后帧并行 + deadline监控 | P1 |
| FR-208 | 动态批处理 | 多传感器源/多ROI动态组batch推理 | P2 |

### FR-300: 专用算子库

| 需求编号 | 功能 | 描述 | 优先级 |
|:----------|:-----|:-----|:-------|
| FR-301 | 视觉算子 | Winograd F(2x2,3x3)/F(4x4,3x3) + FlashAttention + Im2Col + MHA + SiLU/GELU + SE + SPPF | P0 |
| FR-302 | 点云算子 | Voxel化 + 3D稀疏卷积 + PointPillars + BEV池化(LSS) + 3D NMS | P0 |
| FR-303 | BEV 变换 | 多相机BEV特征变换(LSS/Cross-Attention/IPM/Transformer) + 体素池化 + 时间融合 | P1 |
| FR-304 | 轨迹优化算子 | 碰撞检测(AABB/球体/胶囊体) + B-Spline/Catmull-Rom + QP求解器(ADMM+内点法) | P1 |
| FR-305 | 信号处理算子 | RFFT/STFT/Welch PSD + IIR/FIR/中值/Savgol滤波器 + 卡尔曼/EKF + 峰值检测 | P1 |

## 3. 非功能性需求 (NFR)

### 性能约束

| 指标 | 目标值 | 测量方法 | 适用场景 |
|:-----|:-------|:---------|:---------|
| 视觉推理延迟 | < 5ms (端到端) | ONNX模型→qoomodel→infer 延迟 | 目标检测/分割 |
| 控制回路延迟 | < 1ms | 关节控制命令生成 | 运动控制 |
| INT8 推理吞吐量 | > 1000 FPS (ResNet-50) | 批量推理基准 | 视频流处理 |
| 模型加载时间 | < 500ms (100MB .qoomodel) | 冷启动加载 | 技能切换 |
| 内存占用 | < 2GB (含3个并行模型) | RSS 监控 | 端侧 2-16GB 内存 |
| 功耗增量 | < 5W (NPU满载推理) | 功耗计 | 电池续航 |

### 兼容性

| 指标 | 目标值 |
|:-----|:-------|
| ONNX opset 支持 | 11-20 |
| .qoomodel 格式兼容 | 向前兼容2个大版本 |
| 芯片覆盖 | ≥14 款主流芯片 |
| 操作系统 | Linux (aarch64/x86_64), QNX, Android NDK |

## 4. 用户角色

| 角色 | 描述 | 核心需求 |
|:-----|:-----|:---------|
| **模型开发者** | 训练模型并部署到机器人 | ONNX导入、量化编译、性能剖析 |
| **机器人应用开发者** | 使用 qoocore API 加载和运行模型 | 统一推理API、多模型管理、性能监控 |
| **硬件厂商** | 适配 qoocore 到新芯片 | NPU HAL 后端开发、算子注册、芯片支持矩阵 |
| **运维管理员** | 管理设备上的模型 | OTA更新、版本管理、健康监控 |

## 5. 约束条件

| 约束 | 描述 |
|:-----|:-----|
| **技术栈** | C++17 + CMake 3.20+ + ONNX Runtime 1.x |
| **构建目标** | aarch64-linux-gnu, x86_64-linux-gnu, armv7l-linux-gnueabihf |
| **依赖关系** | 被 qoobrain 依赖（提供推理加速）、依赖 qoobody 硬件规范 |
| **实时性** | 支持 Linux PREEMPT_RT 实时内核 |
| **许可** | Apache 2.0 (核心), 厂商SDK按各自许可 |
