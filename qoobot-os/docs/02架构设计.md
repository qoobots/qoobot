> 版本：v0.1 | 最后更新：2026-06-30 | 状态：Draft

## 1. 设计目标与原则

### 1.1 核心设计目标

| 目标 | P0 指标 | P1 目标 | 说明 |
|------|---------|---------|------|
| 端到端推理延迟 | < 10ms (视觉) / < 100ms (VLA) | < 5ms / < 50ms | 芯片级 AI 加速 |
| 控制回路频率 | > 1kHz (关节) / > 100Hz (全身) | > 2kHz / > 200Hz | 实时运动控制 |
| 系统启动时间 | < 30s (冷启动) | < 15s | POST 自检 + 服务初始化 |
| 内存占用 | < 8GB (全部运行时) | < 4GB | 端侧内存约束 |
| 多模型并发 | > 30 FPS (检测+分割+规划) | > 60 FPS | 多任务并行推理 |
| 故障恢复时间 | < 5s (单关节) / < 30s (全系统) | < 2s / < 10s | 降级运行 |
| 电池续航 | > 2h (标准工况) | > 4h | DVFS 功耗优化 |

### 1.2 设计原则

- **分层解耦**：AI 推理层 (QooCore) → 认知决策层 (QooBrain) → 硬件抽象层 (QooBody HAL) → 系统服务层 (QooSvc)，层间通过明确接口通信
- **芯片优先**：AI 推理引擎从第一天起就针对多异构芯片设计，不绑定单一硬件平台
- **实时优先**：控制回路（运动/安全）运行在硬实时优先级，AI 推理运行在软实时优先级，交互服务运行在普通优先级
- **离线优先**：核心能力（语音唤醒、基础导航、本地推理）即使无网络也必须可用
- **安全内置**：硬件安全控制器独立于主控 SoC 运行，软件层无法绕过硬件安全回路
- **技能可组合**：系统服务通过统一 API 暴露，上层技能可自由组合调用
- **仿真即真实**：MuJoCo 仿真环境与真实硬件共享相同的 HAL 接口和控制算法代码

### 2.3 IPC 通信矩阵

| 链路 | 协议 | 传输 | 优先级 | 带宽 |
|------|------|------|:------:|------|
| QooBody HAL → QooCore | DMA-BUF / shared memory | 零拷贝 | RT | > 10 Gbps |
| QooCore → QooBrain | shared memory (ROS 2 topic) | 零拷贝 | RT | > 5 Gbps |
| QooBrain → QooBody HAL | gRPC / ROS 2 action | Unix domain socket | RT | > 1 Gbps |
| QooBrain ↔ QooSvc | gRPC | Unix domain socket | Normal | > 100 Mbps |
| QooSvc → QooStore | in-process (C++) | 直接调用 | Normal | N/A |
| QooBrain → External Apps | gRPC / WebSocket | TCP socket | Normal | > 10 Mbps |
| Safety Loop | CAN FD (独立硬件) | 物理总线 | Hard RT | > 1 Mbps |

## 4. 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 构建系统 | CMake Superbuild + colcon + setuptools | C++/Python/ROS2 混合项目最广泛支持 |
| 层间通信 | gRPC + shared memory + ROS 2 | gRPC 用于服务调用，DMA-BUF 用于数据管道 |
| 模型格式 | FlatBuffers (.qoomodel) + ONNX | FlatBuffers 零拷贝反序列化，ONNX 业界标准 |
| 仿真引擎 | MuJoCo (主) + Gazebo (ROS2 生态) | MuJoCo 高性能物理，Gazebo ROS2 生态兼容 |
| 控制求解器 | qpOASES (QP) + ADMM + 内点法 | 实时 QP 求解最优选择 |
| 安全架构 | 独立安全 MCU + 冗余回路 | 功能安全认证要求，主控 SoC 不可信 |
