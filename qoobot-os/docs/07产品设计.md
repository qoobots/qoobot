## 1. 产品定义

### 1.1 产品定位

qoobot-os 是 QooBot 仿生人的 **"神经系统+大脑"** —— 它是运行在机器人端侧 SoC 上的完整操作系统层，提供从芯片级 AI 加速到高层行为决策的完整能力栈。

**一句话定位**：仿生人版的 Android + ROS 2，为具身智能提供"操作系统级"基础设施。

### 1.2 目标用户

| 用户角色 | 需求 | 使用方式 |
|----------|------|----------|
| 机器人应用开发者 | 快速开发技能，无需关心底层硬件 | Brain SDK (Python) + 技能市场 |
| 模型开发者 | 将 AI 模型部署到端侧芯片 | ai-engine/ CLI + 编译工具链 |
| 硬件厂商 | 将自家传感器/执行器接入 QooBot 生态 | hal/ 驱动接口 |
| 系统集成商 | 定制机器人整机方案 | 配置化参数调整 + 硬件选型指南 |
| 科研人员 | 机器人算法研究与验证 | MuJoCo 仿真 + brain_sim |
| 运维人员 | 机器人健康监控与远程管理 | CLI + 诊断面板 + OTA |

### 1.3 竞品对比

| 维度 | qoobot-os | ROS 2 | NVIDIA Isaac | Android Things |
|------|-----------|-------|-------------|----------------|
| AI 推理 | 多芯片 NPU/GPU/DSP/CPU 统一 | 无内置 AI 引擎 | CUDA/TensorRT | NNAPI |
| 实时控制 | 1kHz RT + SIL2 安全 | RT 支持有限 | 依赖外部 | 非 RT |
| 硬件抽象 | 仿生人专用 HAL (30 DOF+) | 通用传感器 | 机器人基础 | 通用 IoT |
| 技能市场 | 内置边缘运行时 | 无 | 无 | Play Store |
| 仿真 | MuJoCo + Gazebo 双引擎 | Gazebo | Isaac Sim | 无 |
| 离线能力 | 核心功能离线可用 | 依赖网络 | 部分离线 | 依赖云 |
| 授权 | Apache 2.0 开源 | BSD 开源 | 部分闭源 | AOSP 开源 |

```
qoobot-os 部署形态
├── 形态A: 旗舰版 (Qualcomm 8295+)

│   ├── 基础AI: 检测/分割/规划 + 语音 + 导航
│   └── 典型场景: 工厂巡检/仓储搬运

    ├── CPU推理 (NEON/AVX优化)
    ├── 轮式底盘 + 6 DOF 机械臂
    ├── 轻量AI: 检测 + 基础导航
    └── 典型场景: 教育/桌面机器人
```

## 3. 核心价值主张

### 3.1 对应用开发者

> "用 50 行 Python 代码创建一个抓取技能，无需了解 CAN 总线或 NPU 寄存器"

- **Brain SDK** 提供场景级 API (`pick_and_place`, `follow_person`, `patrol`)
- **技能市场** 分发和安装技能，类 App Store 体验
- **仿真环境** 无需真实硬件即可开发测试

### 3.2 对模型开发者

> "将 PyTorch 模型一键编译为芯片优化格式，自动选择最优后端"

- **ai-engine/ 编译链** ONNX → IR → 量化 → .qoomodel
- **多芯片支持** 一次编译，多平台运行 (同架构)
- **性能剖析** 逐层延迟 + 精度对比

### 3.3 对硬件厂商

> "实现 3 个 HAL 接口方法，你的传感器就能接入 QooBot 生态"

- **hal/** 标准化传感器/执行器/通信接口
- **即插即用** 设备自动发现与驱动加载
- **参考设计** 开源硬件参考降低接入成本

### 3.4 对系统集成商

> "通过 YAML 配置文件定制机器人形态，无需修改代码"

- **参数化配置** DOF/关节限位/传感器/执行器均可配置
- **形态因子库** 30/23/20 DOF 仿生人 + 轮式 + 四足
- **安全认证** 硬件安全回路已通过 SIL2 设计审查
