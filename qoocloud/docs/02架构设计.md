# 02 — 架构设计

> 版本：v0.1 | 最后更新：2026-03-17 | 状态：Draft
> 子项目：qoocloud（云端服务）| 架构：Spring Cloud 微服务集群
> 对标：AWS IoT Core + OpenAI API + GitHub Actions OTA 模式

---

## 1. 设计目标与原则

### 1.1 核心设计目标

| 目标 | P0 指标 | P1 目标 | 说明 |
|------|---------|---------|------|
| API 网关可用性 | 99.95% | 99.99% | 云端服务入口 SLA |
| 推理 API 延迟 | < 2s (P50) | < 1s (P50) | 远程大模型推理延迟 |
| 设备在线容量 | 100,000 | 1,000,000 | 同时在线设备数 |
| OTA 升级成功率 | > 99.5% | > 99.9% | 固件/模型/技能升级 |
| 数据同步延迟 | < 5s | < 1s | 端→云数据上传延迟 |
| 推理吞吐 | 1,000 QPS | 10,000 QPS | 推理 API 并发处理 |

### 1.2 设计原则

- **端云协同**：重型推理上云、轻量推理在端，无缝切换
- **设备即代码**：所有机器人配置、策略、技能版本化，可追溯、可回滚
- **弹性伸缩**：按设备数量、推理负载自动扩缩容
- **安全优先**：所有通信链路 mTLS 加密、设备身份强验证、多租户数据隔离
- **可观测性驱动**：全链路追踪、指标监控、智能告警贯穿所有服务
- **渐进式交付**：先核心推理+设备管理，再 OTA+数字孪生，最后多机器人编排

## 3. 技术选型

### 3.1 语言与框架

| 组件 | 技术选型 | 理由 |
|------|---------|------|
| 语言 | Java 21 LTS | 虚拟线程、模式匹配、密封类 |
| Web 框架 | Spring Boot 3.3 + Spring MVC | 成熟稳定，生态丰富 |
| 微服务框架 | Spring Cloud 2024.x + Spring Cloud Alibaba | Nacos 注册/配置、Sentinel 熔断限流 |
| API 网关 | Spring Cloud Gateway | 统一入口、路由、鉴权、限流 |
| 服务间调用 | OpenFeign + gRPC | REST 同步 + gRPC 高性能内部调用 |
| 消息队列 | RocketMQ | 异步事件驱动、高吞吐、事务消息 |
| ORM | MyBatis-Plus + Spring Data JPA | 复杂查询 + 标准 CRUD |
| 数据库迁移 | Flyway | Java 生态标准数据库版本管理 |
| 数据库 | PostgreSQL 16 + ClickHouse | 事务型 + 分析型 |
| 缓存 | Redis 7 Cluster | 分布式缓存、会话、排行榜 |
| 对象存储 | MinIO | S3 兼容，模型/固件/日志存储 |
| 推理引擎 | Triton Inference Server + vLLM | GPU 推理调度 + 大模型服务 |
| 全链路追踪 | Apache SkyWalking | 端到端追踪，与 Spring Cloud 深度集成 |
| 日志聚合 | ELK (Elasticsearch + Logstash + Kibana) | 业界标准日志平台 |
| 指标监控 | Prometheus + Grafana | 时序指标 + 可视化仪表板 |
| 容器编排 | Kubernetes | 弹性伸缩、滚动更新、服务发现 |
| CI/CD | GitHub Actions + ArgoCD | GitOps 工作流 |
| API 文档 | SpringDoc OpenAPI 3.0 | 自动生成 REST API 文档 |

### 3.2 微服务拆分原则

| 服务 | 拆分理由 | 独立扩展原因 |
|------|---------|------------|
| inference | GPU 密集型 | 推理负载波动大，需独立 GPU 节点池 |
| device | IO 密集型 | 设备长连接 (WebSocket) 与短连接 API 混合 |
| ota | 带宽密集型 | 大文件分发，需 CDN + 独立带宽 |
| data | 存储密集型 | 数据管道 + ETL，需独立存储扩容 |
| orchestra | CPU 密集型 | 调度算法计算密集 |
| twin | GPU 密集型 | 仿真渲染需要 GPU |
| observability | 基础设施 | 独立于业务服务，保证可观测 |
| infra | 基础设施 | 跨切面服务 |

## 4. 核心数据流

### 4.1 设备上线流程

```
[QooBot 设备] → [qoobot-cloud-gateway] → [qoocloud-device]
                                               │
                                               ├─ 1. 验证 JWT/设备证书 → qooauth
                                               ├─ 2. DeviceService.registerHeartbeat()
                                               │      └─ Redis (设备在线状态)
                                               ├─ 3. 检查 OTA 更新 → qoocloud-ota
                                               │      └─ 如果有待更新 → 推送通知
                                               ├─ 4. 同步配置 → ConfigService
                                               │      └─ 下发最新机器人参数
                                               └─ 5. 记录事件 → RocketMQ
                                                      └─ 可观测性服务消费
```

### 4.2 远程推理流程

```
[设备请求推理] → Gateway → qoocloud-inference
                               │
                               ├─ 1. 认证鉴权 → qooauth
                               ├─ 2. 端云混合路由器
                               │      ├─ 简单任务 → 返回建议端侧推理
                               │      └─ 复杂任务 → 云端推理
                               ├─ 3. 推理调度器
                               │      ├─ GPU 资源分配
                               │      ├─ 排队管理（优先级队列）
                               │      └─ 模型热加载（如需要）
                               ├─ 4. 推理引擎 (Triton/vLLM)
                               │      └─ 流式返回 (SSE/gRPC stream)
                               ├─ 5. 推理缓存写入
                               │      └─ Redis (语义相似去重)
                               └─ 6. 审计记录
                                      └─ Token 用量 + 延迟 + 成本
```

### 4.3 OTA 升级流程

```
[运维人员/自动触发] → qoocloud-ota
                         │
                         ├─ 1. 创建升级任务
                         │      ├─ 上传固件/模型/技能包 → MinIO
                         │      ├─ 选择目标设备/分组
                         │      └─ 配置灰度策略
                         │
                         ├─ 2. 灰度发布
                         │      ├─ 第 1 批: 5% 设备
                         │      ├─ 监控成功率 + 异常指标
                         │      ├─ 第 2 批: 25% 设备
                         │      ├─ 再次监控
                         │      └─ 全量发布
                         │
                         ├─ 3. 设备端执行
                         │      ├─ 下载增量包 (delta)
                         │      ├─ 校验签名
                         │      ├─ 安装/替换
                         │      └─ 重启服务
                         │
                         └─ 4. 回滚机制
                                ├─ 设备上报升级失败
                                ├─ 自动回滚到上一版本
                                └─ 告警通知运维
```
