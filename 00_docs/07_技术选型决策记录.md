### ADR-001：ROS 2 版本选择

| 属性 | 内容 |
|------|------|
| **ADR-ID** | ADR-001 |
| **状态** | ⚡ 建议 |
| **优先级** | P0 |
| **关联** | D01, D09, D13, D37 |

**Context**：现有文档（01/03）写 "ROS 2 Humble/Iron" 或 "ROS 2 Humble"。Humble 是 LTS（支持至 2027 年 5 月），Iron 已 EOL。Jazzy Jalisco 是当前 LTS（支持至 2029 年 5 月）。Phase 1 启动时如选错版本，半年后可能面临强制升级。

**Decision**：

> **选择 ROS 2 Jazzy Jalisco（LTS）**

| 排序列 | 方案 | 评估 |
|--------|------|------|
| ✅ 选中 | **ROS 2 Jazzy** | LTS 至 2029/05，Ubuntu 24.04 官方支持，最新生态包兼容 |
| ❌ 淘汰 | ROS 2 Humble | LTS 至 2027/05，仅一年半后 EOL，Ubuntu 22.04 绑定 |
| ❌ 淘汰 | ROS 2 Iron | 已 EOL |
| ❌ 淘汰 | ROS 2 Rolling | 滚动发布，API 不稳定 |

**Rationale**：
1. Jazzy 是当前最新 LTS，2029 年 5 月才 EOL，覆盖开发 + 早期部署全周期
2. 生态包（MoveIt 2、Navigation 2、ros2_control）对 Jazzy 支持已成熟
3. Ubuntu 24.04 官方仓库直接支持，无需 PPA
4. 若选 Humble，2027 年中需强制升级，届时业务代码已积累，迁移成本高

**Consequences**：
- ✅ 长期稳定性保证（4+ 年 LTS 覆盖）
- ✅ 避免 Phase 2/3 强制升级风险
- ⚠️ 部分社区教程仍基于 Humble，需适配
- ⚠️ 测试与 CI 需升级 Ubuntu 22.04 → 24.04（Docker 镜像）
- ⚠️ Isaac Sim / Gazebo Ignition 的 ROS 2 Jazzy bridge 需验证兼容性

**Action**：
1. 验证 MoveIt 2 / Navigation 2 / ros2_control 对 Jazzy 的支持状态
2. 更新 Docker 基础镜像至 `ubuntu:24.04` + ROS 2 Jazzy
3. 更新 03_系统架构设计.md 中 ROS 2 版本引用

### ADR-002：DDS 实现选择

| 属性 | 内容 |
|------|------|
| **ADR-ID** | ADR-002 |
| **状态** | ⚡ 建议 |
| **优先级** | P0 |
| **关联** | ADR-001, D37 |

**Context**：文档 01/03 同时列出了 Fast-DDS 和 Cyclone DDS，但未作选择。两者均可作为 ROS 2 底层 DDS 实现，但性能特征和稳定性有显著差异。

**Decision**：

> **选择 Eclipse Cyclone DDS（默认）+ Fast-DDS（备选）**

| 排序列 | 方案 | 评估 |
|--------|------|------|
| ✅ 选中 | **Cyclone DDS** | ROS 2 官方推荐默认 RMW（从 Jazzy 起）、零拷贝传输、Eclipse 基金会维护 |
| ⚠️ 保留 | Fast-DDS | eProsima 维护、更丰富 QoS、但内存占用和复杂场景下稳定性不如 Cyclone |
| ❌ 淘汰 | Connext DDS | 商业许可 |

**Rationale**：
1. ROS 2 Jazzy 已将 Cyclone DDS 作为 **默认 RMW**（`RMW_IMPLEMENTATION=rmw_cyclonedds_cpp`）
2. Cyclone DDS 在延迟、吞吐量和内存占用三项关键指标上均优于 Fast-DDS（参考 ROS 2 官方 Benchmark）
3. 零拷贝共享内存传输（iceoryx 集成），同机进程间通信延迟可降至 < 10μs——机器人实时控制刚需
4. Eclipse 基金会维护，社区活跃，发布节奏稳定

**Alternative Considerations**：
- **Fast-DDS** 在需要以下特性时保留：丰富的 QoS Profile（Partition）、DDS Security（mTLS）、XML 配置更加灵活。保留作为备选，当 Cyclone DDS 在特定场景下有兼容性问题时切换。

**Consequences**：
- ✅ 更低延迟、更低内存、官方推荐
- ✅ 与 ROS 2 Jazzy 默认配置一致，减少配置工作
- ⚠️ Fast-DDS 作为备选需要 CI 中双 RMW 编译验证

**Action**：
1. Docker 环境默认使用 `rmw_cyclonedds_cpp`
2. CI 增加 Cyclone DDS + Fast-DDS 双 RMW 编译测试
3. 如使用共享内存传输，评估 iceoryx 集成可行性

### ADR-003：深度学习框架

| 属性 | 内容 |
|------|------|
| **ADR-ID** | ADR-003 |
| **状态** | ⚡ 建议 |
| **优先级** | P0 |
| **关联** | D01-D06, D09-D12 |

**Context**：所有文档隐含使用 PyTorch（ONNX Runtime、TensorRT-LLM、3DGS 等均基于 PyTorch 生态），但未明确声明。这影响所有模型训练、微调、推理管线。

**Decision**：

> **确认 PyTorch 2.x 为唯一深度学习框架**

| 排序列 | 方案 | 评估 |
|--------|------|------|
| ✅ 选中 | **PyTorch 2.x** | 与所有已选模型/Qwen/OpenVLA/3DGS/ONNX/TensorRT 原生集成 |
| ❌ 淘汰 | JAX | 生态与已选模型严重不兼容，OpenVLA/YOLOv11 无 JAX 实现 |
| ❌ 淘汰 | TensorFlow/Keras | 机器人领域边缘化，3DGS 不支持 |

**Rationale**：
1. Qwen2.5 系列、OpenVLA、YOLOv11、SAM 2、FoundationPose、3DGS 均以 PyTorch 为首要框架
2. ONNX Runtime 导出、TensorRT-LLM 编译均以 PyTorch 模型为入口
3. Isaac Sim / Isaac Lab 训练管线基于 PyTorch
4. 选 PyTorch 是 "自然选择" 而非 "决策"——所有依赖已指向它

**Consequences**：
- ✅ 零额外适配成本，所有已选组件原生支持
- ⚠️ 需明确版本约束（PyTorch 2.2+ for `torch.compile`，2.4+ for 某些新特性）
- ⚠️ 与 TensorRT-LLM 版本适配矩阵需记录

**Action**：
1. 在 `pyproject.toml` / `requirements.txt` 中锁定 PyTorch 版本
2. 建立 PyTorch → TensorRT-LLM → ONNX 版本兼容矩阵文档

### ADR-004：Qwen2.5 模型部署尺寸确定

| 属性 | 内容 |
|------|------|
| **ADR-ID** | ADR-004 |
| **状态** | ⚡ 建议 |
| **优先级** | P0 |
| **关联** | D01, D03 |

**Context**：文档选定 Qwen2.5-7B 作为边缘主模型，但 Jetson Orin AGX 64GB 内存勉强可跑 7B INT4（约占用 4-5GB VRAM），Nano/低配 Orin NX 16GB 版则非常紧张。需拉取多个尺寸进行实测。

**Decision**：

> **阶段性确认：7B INT4 直出（目标），同步准备 3B/1.5B 降级路径**

| 场景 | 模型 | 硬件 | 推理延迟目标 |
|------|------|------|-------------|
| 目标主模型 | Qwen2.5-7B-Instruct INT4 | Jetson Orin AGX 64GB | < 500ms |
| 降级方案 A | Qwen2.5-3B-Instruct INT4 | Jetson Orin NX 16GB | < 300ms |
| 降级方案 B | Qwen2.5-1.5B-Instruct INT4 | CPU / 低配 Orin | < 200ms（非 GPU） |

**Rationale**：
1. 7B 在 INT4 量化下推理延迟与 3B FP16 相当，但指令遵循和逻辑推理能力显著更强
2. 真实硬件测试是唯一可靠的判断标准，Phase 1 Sprint 2 第一步就是实测延迟/内存
3. 软件抽象层（`llm_agent` 的 `ModelLoader` 接口）设计为多尺寸通用，切换尺寸无需改代码

**Consequences**：
- ✅ 架构层面支持多尺寸热切换
- ⚠️ 硬件采购需确保至少有 Orin AGX 64GB 用于开发
- ⚠️ 如果 7B 延迟不达标，Sprint 2 中期可能切换到 3B

**Action**：
1. Sprint 2 T2.1 任务中增加：7B/3B/1.5B 三种尺寸的延迟/内存 Benchmark
2. 基于实测数据锁定最终尺寸
3. 无论选择哪个尺寸，推理服务接口保持一致

### ADR-005：语音识别与合成（ASR/TTS）

| 属性 | 内容 |
|------|------|
| **ADR-ID** | ADR-005 |
| **状态** | ⚡ 建议 |
| **优先级** | P0 |
| **关联** | 交互上下文 |

**Context**：02/03/05 文档中多处涉及语音输入输出（"语音对话"、"用户语音 → 文本"、"SpeakAction"），但从未指定 ASR/TTS 方案。这是自然语言交互的入口通道。

**Decision**：

> **Phase 1：Whisper（ASR）+ Edge-TTS / Coqui TTS（TTS）**
> **Phase 2：评估自研语音模型（可选）**

| 组件 | Phase 1 选择 | 理由 |
|------|-------------|------|
| ASR（语音→文本） | **Whisper** (openai-whisper 或 faster-whisper) | 多语言（含中文）最强开源 ASR，ONNX 可部署，支持 tiny→large 多尺寸 |
| TTS（文本→语音） | **Edge-TTS**（优先，零部署成本）+ **Coqui TTS**（备选，离线） | Edge-TTS 调用微软免费 API（中文效果好）；Coqui TTS（原 Mozilla TTS）完全离线 |

**Rationale**：
1. **Whisper**：
   - 中文识别 WER 领先，50+ 语言原生支持
   - `faster-whisper`（CTranslate2 后端）在 Jetson 上 large-v3 模型可达实时
   - whisper.cpp 版本可在纯 CPU 上运行
2. **Edge-TTS**：
   - 零部署——调用微软 Edge 免费 TTS API
   - 中文语音自然度极高，支持多音色
   - Phase 1 无需训练/部署任何 TTS 模型
3. **Coqui TTS**（备选）：
   - 离线可用，不依赖外部 API
   - 中文预训练模型可用（XTTS v2）
   - 部署较重（需 GPU），作为降级方案

**Alternative Considerations**：
- **Whisper 尺寸选择**：Phase 1 推荐 `faster-whisper-large-v3`，差环境切 `whisper.cpp small`
- **自研 TTS**：不应在 Phase 1 启动，延迟到 Phase 2+ 按需评估

**Consequences**：
- ✅ Phase 1 零成本获得高质量中文 ASR/TTS
- ⚠️ Edge-TTS 依赖外部 API（离线不可用），Coqui TTS 需作为离线降级方案
- ⚠️ Whisper 在 Jetson 上的实时性能需实测

**Action**：
1. Sprint 2 增加 Whisper 集成任务（T2.8）
2. 评估 faster-whisper 在目标 Jetson 上的推理延迟
3. 设计 `voice_io` 模块（ASR → IntentParser → TTS 管道），与 LLM agent 松耦合

### ADR-007：gRPC-Web 代理

| 属性 | 内容 |
|------|------|
| **ADR-ID** | ADR-007 |
| **状态** | ⚡ 建议 |
| **优先级** | P0 |
| **关联** | D37, D41, 交互上下文 |

**Context**：架构设计（03 §4.3）中 brain_viz 需要与 brain_ai/brain_core 通过 gRPC-Web 通信。浏览器不能直接发 gRPC 请求，需要代理转换 gRPC-Web → gRPC。文档 03 提到 Envoy 但未明确选定。

**Decision**：

> **选择 Envoy 作为 gRPC-Web 代理**

| 排序列 | 方案 | 评估 |
|--------|------|------|
| ✅ 选中 | **Envoy** | gRPC-Web 官方推荐代理、CNCF 毕业项目、配置成熟 |
| ❌ 淘汰 | grpcwebproxy（improbable） | 已停止维护 |
| ❌ 淘汰 | Nginx + 自定义模块 | 配置复杂、非官方路径 |

**Rationale**：
1. gRPC-Web 官方文档明确推荐 Envoy
2. Envoy 已 Docker 化，一条命令即可代理 gRPC-Web → gRPC 双向转换
3. CNCF 毕业项目，生产级稳定性
4. gRPC-Web + Envoy 支持 `text/event-stream` 和 `application/grpc-web+proto` 两种前端调用方式

**Consequences**：
- ✅ 一行 Docker Compose 配置即可启动
- ✅ 与 Next.js 开发服务器兼容（独立端口 8080）

### ADR-008：物理引擎选择

**Consequences**：
- ✅ 仿真训练→验证环境物理引擎一致
- ⚠️ GPU 不可用时需降级 Bullet
- ⚠️ PhysX Python 包生态不如 PyBullet 成熟，需验证 API 稳定性

**Action**：
1. Phase 2 世界模型物理验证模块开发前安装 `physx` Python 包并评估 API 覆盖度

### ADR-014：坐标系统约定

| 属性 | 内容 |
|------|------|
| **ADR-ID** | ADR-014 |
| **状态** | ⚡ 建议 |
| **优先级** | P1 |
| **关联** | 所有感知/控制/可视化模块 |

**Context**：机器人领域坐标系约定混乱：ROS 标准使用右手坐标系（X 前 Y 左 Z 上），而某些视觉/3D 引擎可能使用不同约定（如 OpenGL Y 上、Unreal Z 上）。需统一约定避免全栈坐标系混乱。

**Decision**：

> **统一采用 ROS 标准坐标系（右手）：X 前（forward）、Y 左（left）、Z 上（up）**
> **前端 Three.js 适配层：所有传入数据转换为 Y 上坐标系（Three.js 默认）**

| 场景 | 坐标系 | 说明 |
|------|--------|------|
| ROS 2 内部 | X 前 / Y 左 / Z 上（右手） | ROS 标准、所有 Topic 消息以此为准 |
| MoveIt 2 运动规划 | 同上 | MoveIt 2 原生 ROS 坐标系 |
| 3DGS 重建 | 同上 | 与 SLAM/map 坐标系统一 |
| Three.js 渲染 | Y 上（WebGL 默认） | `+Z→+Y, +Y→-Z` 转换适配 |

**Rationale**：
1. ROS 2 生态所有工具（RViz、Foxglove、MoveIt 2、Navigation 2）均使用 ROS 右手坐标系
2. 强制所有内部通信使用 ROS 坐标系——避免 C++ ↔ Python 间的坐标转换
3. 仅在脑 visual 端（Three.js）做坐标系适配——且只在一个 Adapter 层做

**坐标转换矩阵（参考）**：
```
ROS → Three.js:
  [x, y, z]_ros → [x, z, -y]_threejs
  即：X→X, Y→Z, Z→-Y
```

**Consequences**：
- ✅ 内部零坐标转换，性能和一致性最优
- ✅ 前端只需一个 `ros2three()` 工具函数
- ⚠️ 如果有外部工具使用不同坐标系（如 UE），需额外适配

**Action**：
1. 在 README 和开发规范中明确坐标系约定
2. 前端封装 `ros2three(point: Point3D): THREE.Vector3` 工具函数
3. URDF 模型加载时验证坐标系一致性

### ADR-015：数值计算加速

| 属性 | 内容 |
|------|------|
| **ADR-ID** | ADR-015 |
| **状态** | ⚡ 建议 |
| **优先级** | P1 |
| **关联** | ADR-003（PyTorch 确认） |

**Context**：Python 端（brain_ai）需要做大量 3D 变换、点云处理、矩阵运算。NumPy 是默认选择，但是否需要 CuPy / JAX 加速这些非模型推理的计算？

**Decision**：

> **Phase 1-2：NumPy + PyTorch（利用已有 GPU 张量）**
> **不引入 CuPy / JAX**

| 排序列 | 方案 | 评估 |
|--------|------|------|
| ✅ 选中 | **NumPy + PyTorch 张量** | PyTorch GPU 张量可替代 CuPy、无需引入额外依赖 |
| ❌ 淘汰 | CuPy | API 与 NumPy 兼容但调试困难，PyTorch 张量已覆盖 GPU 计算 |

**Rationale**：
1. Brain OS 已依赖 PyTorch（ADR-003），PyTorch 张量天然支持 GPU 加速
2. NumPy ↔ PyTorch Tensor 转换零拷贝（`torch.from_numpy()`）
3. 3D 变换矩阵、点云处理用 PyTorch 张量在 GPU 上计算，性能与 CuPy 相当
4. 减少一个大型依赖（CuPy 约 500MB）

**Consequences**：
- ✅ 零额外 GPU 加速依赖
- ⚠️ 需团队成员习惯使用 PyTorch 而非 CuPy 做 GPU 数值计算

**Action**：
1. 确立规范：GPU 上的数值计算用 PyTorch Tensor，CPU 上用 NumPy
2. 封装常用 3D 变换工具函数（`brain_ai/utils/transforms.py`）
