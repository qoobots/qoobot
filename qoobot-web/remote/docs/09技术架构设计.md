| 层次 | 技术选型 | 用途 |
|:-----|---------|------|
| 实时通信 | WebRTC (libwebrtc / GStreamer) | 音视频+数据通道 |

| 机器人端 | C++17 + GStreamer / libwebrtc | 编码/解码/传输 |
| 云端 | Java 17 + Spring Boot + Netty | 信令中继/会话管理 |
| Web 操控端 | TypeScript + Vue 3 + WebRTC API | 浏览器内操控 |
| VR 操控端 | Unity / WebXR | VR 沉浸式操控 |
| 桌面操控端 | Python + PySide6 + aiortc | 桌面应用操控 |

## 2. 关键技术决策

### 2.1 为什么选择 WebRTC

| 对比 | WebRTC | 自定义 UDP | RTMP/HLS |
|:-----|:-----:|:---------:|:------:|
| 端到端延迟 | <100ms ✅ | <50ms | 2-5s |
| NAT 穿透 | ICE/STUN/TURN ✅ | 需自建 | 无 |
| 加密 | DTLS/SRTP 内置 ✅ | 需自建 | 可选 |
| 浏览器支持 | 原生 ✅ | 需插件 | 原生 |
| P2P 传输 | 支持 ✅ | 支持 | 不支持 |

**决策**：WebRTC 作为主要传输协议，提供低延迟、加密、NAT 穿透和跨平台能力。

### 2.2 视频编码策略

```
场景检测:
  静态场景 → 降低帧率 (15fps) + 提高质量
  运动场景 → 提高帧率 (30fps) + 适度质量
  带宽不足 → 降低分辨率 + 降低帧率
  
自适应码率 (ABR):
  基于 RTCP 反馈的丢包率、RTT 动态调整
```

### 2.3 传输优先级

```
QoS 优先级:
  1. 控制指令: 最高优先级, 可靠传输 (DataChannel ordered)
  2. 安全信号: 次高优先级, 独立心跳通道
  3. 音频: 低延迟, 可容忍少量丢包 (FEC)
  4. 视频: 尽力传输, 自适应降质
  5. 传感器: 后台传输, 不影响操控体验
```

## 3. 延迟预算

### 3.1 局域网场景 (目标<100ms)

```
采集 (10ms) → 编码 (15ms) → 网络 (5ms) → 
解码 (10ms) → 渲染 (10ms) → 输入 (5ms) → 
网络 (5ms) → 执行 (10ms)
─────────────────────────────────
总计: ~70ms (留30ms余量)
```

### 3.2 互联网场景 (目标<200ms)

```
采集 (10ms) → 编码 (15ms) → 网络 (30ms) → 
解码 (10ms) → 渲染 (15ms) → 输入 (10ms) → 
网络 (30ms) → 执行 (10ms)
─────────────────────────────────
总计: ~130ms (留70ms余量)
```

## 4. 安全传输

```
WebRTC 安全层次:
  ┌────────────────────────────┐
  │ DTLS-SRTP (媒体加密)       │
  │ AES-128-GCM 默认           │
  ├────────────────────────────┤
  │ SCTP over DTLS (数据加密)  │
  │ DataChannel 可靠/不可靠    │
  ├────────────────────────────┤
  │ ICE (NAT穿透)              │
  │ STUN/TURN 中继             │
  ├────────────────────────────┤
  │ Signaling (信令安全)       │
  │ WSS + JWT Token            │
  └────────────────────────────┘
```
