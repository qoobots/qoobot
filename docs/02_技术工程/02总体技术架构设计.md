## 🎯 技术栈收敛策略

### 问题背景

QooBot 是大型人形机器人开源工程，当前已有 13 个子项目，横跨四层架构。已使用 **C++、C、Python、Java、TypeScript/JavaScript** 共 5 种编程语言（不含 HTML/CSS 标记语言）。如果每个子项目引入新语言，将导致：

- **贡献门槛过高**：社区开发者需要掌握过多语言才能参与多个子项目
- **工具链碎片化**：CI/CD、lint、格式化、依赖管理需要维护多套配置
- **代码复用困难**：跨项目的共享库（如 Protobuf 定义、公共工具函数）难以统一
- **维护成本指数增长**：每多一种语言，就多一套生态需要跟进

### 收敛原则

1. **不引入新语言**：已完成技术选型的子项目保持现状；规划中的子项目从已有语言池中选取
2. **场景优先匹配**：每种语言有其最佳场景，选型依据如下：

| 场景 | 首选语言 | 原因 |
|------|----------|------|
| 实时控制、推理引擎、嵌入式固件 | **C++ / C** | 确定性延迟、零成本抽象、硬件直访 |
| AI/ML 训练与推理、数据处理、标定脚本 | **Python** | 生态最丰富（PyTorch/Transformers/OpenCV） |
| 云端微服务、API 网关、认证中间件 | **Java (Spring Cloud)** | 企业级微服务生态（服务发现/配置中心/网关/熔断） |
| Web 前端、IDE 插件、可视化面板 | **TypeScript** | 类型安全 + 前端生态（Vue 3/Vite/Element Plus） |

3. **构建系统统一**：CMake（C++）、Maven/Gradle（Java）、setuptools/pyproject.toml（Python）、npm/pnpm（TypeScript）
4. **通信协议统一**：跨项目通信全部使用 gRPC + Protobuf，避免 REST/GraphQL/MQTT 混用
5. **容器化策略统一**：所有子项目均提供 Dockerfile + Docker Compose / K8s 部署方案

### 语言池（已锁定，不再扩展）

```
C/C++       ── 性能层（控制、推理、固件、仿真）
Python      ── AI 层（训练、推理编排、工具脚本、标定）
Java        ── 服务层（微服务、API 网关、认证、设备管理）
TypeScript  ── 展现层（Web 前端、IDE 插件、可视化）
```

> **红线**：规划中的子项目禁止引入 Rust、Kotlin、Swift、Go、C# 等新语言。确有必要时需通过 RFC 流程论证并取得社区共识。

## 🔐 qooauth — 账号与安全

> **定位**：统一身份基础设施，对标 Apple ID。
> **状态**：设计阶段，待 Java/Spring Cloud 重写。

### 架构模式：Spring Cloud 微服务

qooauth 采用 **Spring Cloud 微服务架构**，按业务域拆分为独立微服务，通过 Spring Cloud Gateway 统一入口：

```
客户端请求
    │
    ▼
┌─────────────────────────────────────────────────┐
│           Spring Cloud Gateway                   │
│   (限流/路由/JWT 预验签/跨域/CORS)               │
└────────┬──────────┬──────────┬──────────────────┘
         │          │          │
    ┌────▼────┐ ┌───▼───┐ ┌───▼────┐
    │ auth    │ │ user  │ │ device │  ... (更多微服务)
    │ 认证服务 │ │ 用户   │ │ 设备   │
    └────┬────┘ └───┬───┘ └───┬────┘
         │          │          │
         └──────────┼──────────┘
              ┌─────▼─────┐
              │ PostgreSQL │
              │   + Redis  │
              └───────────┘
```

## 📐 设计原则

1. **端侧优先**：核心计算在端侧完成，云端作为补充和协同
2. **接口标准化**：跨项目通信统一使用 gRPC + Protobuf
3. **渐进式架构**：新子项目从文档开始，逐步构建代码
4. **技术栈收敛**：锁定 4 种编程语言（C++/Python/Java/TypeScript），规划中的子项目不引入新语言，最大化代码复用与社区参与度
5. **多语言协作**：根据场景选择最优语言（C++ 性能/Java 微服务/Python AI/TypeScript 前端）
6. **可观测性优先**：每个服务内置 metrics/health/logging
7. **安全内建**：认证/加密/审计贯穿所有服务

### 技术栈对齐矩阵

规划中子项目与已实现子项目的技术栈对齐关系：

```
qoobrain (C++/Python/TS) ──对齐──▶ qoosvc (C++/Python)
qooauth (Java/Spring Cloud) ──对齐──▶ qoocloud (Java/Spring Cloud) ──对齐──▶ qoostore (Java/TS)
qoodev (Python/TS)           ──对齐──▶ qoochain (Python) ──对齐──▶ qoogear (Java/TS/C)
qoobrain brain_viz (TS)      ──对齐──▶ qoocommunity (TS)
```

跨项目共享的关键组件：
- **Spring Cloud 微服务治理**：qooauth → qoocloud → qoostore（Nacos 注册/配置、Sentinel 熔断、Seata 分布式事务、SkyWalking 链路追踪）
- **Spring Security 认证体系**：qooauth → qoocloud → qoostore（JWT 过滤器、OAuth2 客户端、权限注解）
- **公共 Maven 模块**：qooauth → qoocloud → qoostore（DTO/VO/工具类/异常码/拦截器）
- **数据库迁移工具**：qooauth → qoocloud → qoostore（基于 Flyway/Liquibase）
- **C HAL 头文件**：qoobody → qoogear（传感器/执行器接口标准）
- **pybind11 绑定模式**：qoocore → qoochain（C++ 算法封装为 Python 可调用）
- **Protobuf 定义**：qoobrain → qoosvc → qoocloud（共享消息类型）
- **前端组件库**：qoobrain brain_viz → qoostore → qoocommunity（共享 UI 组件，基于 Vue 3 + Element Plus）
