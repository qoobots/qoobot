# QooBot 总体质量保障设计文档

> 最后更新：2026-06-29
>
> 本文档定义 QooBot 项目的总体质量保障体系，涵盖测试策略、质量门禁、代码规范、性能基准、安全测试与质量度量。

---

## 🎯 质量目标

| 维度 | 目标 | 度量方式 |
|------|------|----------|
| **代码质量** | 技术债务率 < 5% | SonarQube / CodeClimate |
| **测试覆盖率** | 单元测试 ≥ 80% | JaCoCo (Java) / coverage.py (Python) / gcov (C++) |
| **缺陷率** | 线上缺陷数 < 0.5/千行代码 | 缺陷管理系统 |
| **可用性** | 云端 99.99%，端侧 MTBF > 1000h | 监控系统 |
| **安全漏洞** | 已知高危漏洞 0 | 依赖扫描 + 渗透测试 |
| **性能回归** | 关键路径性能不劣化 > 10% | 基准测试框架 |
| **文档覆盖** | 公开 API 100% 有文档 | 文档自动生成 |

### 提交规范

```
<type>(<scope>): <subject>

类型 (type):
  feat     - 新功能
  fix      - 缺陷修复
  docs     - 文档变更
  style    - 代码格式
  refactor - 重构
  perf     - 性能优化
  test     - 测试相关
  build    - 构建/CI
  chore    - 其他杂项
  security - 安全修复

范围 (scope): 子项目名 (qoobrain/qoocore/qoobody/...)

示例:
  feat(qoobrain): add multi-modal sensor fusion
  fix(qooauth): resolve JWT token refresh race condition
  security(qoocloud): patch unauthorized inference access
```

## ⚡ 性能基准

### 关键性能指标 (KPI)

#### 端侧基准

| 基准测试 | 目标 | 测试方法 |
|----------|------|----------|
| **感知 Pipeline 延迟** | < 50ms (e2e) | 固定场景 + 标准输入 |
| **SLAM 定位精度** | < 5cm RMSE | 标准轨迹 + 真值对比 |
| **LLM 首 Token 延迟** | < 2s (7B) | 标准 Prompt + 本地 GPU |
| **控制闭环频率** | ≥ 1kHz | 空载 + 满载 |
| **模型推理延迟** | 按模型清单 | ONNX Runtime Benchmark |
| **技能冷启动** | < 1s | 标准技能包安装 |
| **系统启动** | < 30s | 冷启动 + 热启动 |

#### 云端基准

| 基准测试 | 目标 | 测试方法 |
|----------|------|----------|
| **API P50** | < 50ms | wrk2 / k6 负载生成 |
| **API P99** | < 500ms | wrk2 / k6 负载生成 |
| **吞吐量** | > 10K QPS (单实例) | 渐进负载 |
| **长连接数** | > 100K WebSocket | 连接压测 |
| **数据库读写** | 读 < 10ms, 写 < 30ms | pgbench |

### 性能回归检测

```
发布前性能测试流程:

  1. 运行基准测试套件
  2. 与上次发布基准对比
  3. 生成性能差异报告
  
  回归判定:
  · 劣化 < 5%  → 自动通过
  · 劣化 5-10% → 需要说明 (可在 PR/MR 中标注)
  · 劣化 > 10% → 阻断发布，需要定位优化
```

## 🔒 安全测试

### 安全测试矩阵

| 测试类型 | 工具 | 频率 | 说明 |
|----------|------|:--:|------|
| **SAST (静态分析)** | SonarQube + CodeQL | 每次 PR | 代码漏洞扫描 |
| **SCA (依赖扫描)** | Dependabot / Trivy | 每日 | 第三方依赖漏洞 |
| **容器扫描** | Trivy / Grype | 每次构建 | 容器镜像漏洞 |
| **密钥检测** | Gitleaks / truffleHog | 每次 PR | 防止密钥泄露 |
| **DAST (动态分析)** | OWASP ZAP | 每周 | 运行时漏洞 |
| **渗透测试** | 人工 + 自动化 | 每季度 | 全面安全评估 |
| **模糊测试** | libFuzzer / AFL | 持续 | 输入边界测试 |
| **安全审查** | 人工 | 每个大版本 | 架构安全审查 |

### 安全测试检查清单

| 检查项 | 说明 |
|--------|------|
| ☐ 认证绕过 | 未授权访问受保护资源 |
| ☐ Token 伪造 | JWT 签名算法混淆攻击 |
| ☐ SQL 注入 | 参数化查询覆盖 |
| ☐ XSS | 输出编码 + CSP 头 |
| ☐ CSRF | CSRF Token + SameSite Cookie |
| ☐ 越权访问 | 横向/纵向权限提升 |
| ☐ 敏感数据泄露 | 日志/响应/错误信息 |
| ☐ 速率限制 | API 滥用防护 |
| ☐ 文件上传 | 类型/大小/扫描 |
| ☐ 加密弱点 | 不安全的算法/密钥长度 |
| ☐ 会话安全 | Token 过期/撤销/轮换 |

## 🔍 代码审查规范

### 审查检查清单

| 类别 | 检查项 |
|------|--------|
| **功能** | ☐ 实现与需求一致 ☐ 边界条件处理 ☐ 错误处理完备 |
| **设计** | ☐ 遵守架构规范 ☐ 无过度设计 ☐ 接口向后兼容 |
| **代码** | ☐ 命名清晰 ☐ 无死代码 ☐ 无硬编码 ☐ 注释清晰 |
| **测试** | ☐ 单元测试覆盖新代码 ☐ 边界/异常测试 ☐ 测试可读 |
| **安全** | ☐ 无 SQL 注入 ☐ 认证授权正确 ☐ 敏感数据保护 |
| **性能** | ☐ 无明显性能问题 ☐ 数据库查询优化 ☐ 缓存策略合理 |
| **文档** | ☐ API 文档更新 ☐ README 更新 ☐ 变更日志更新 |
