# QooBot 总体安全架构设计

> 最后更新：2026-06-29
>
> 本文档定义 QooBot 项目的总体安全架构，涵盖安全分层、身份认证、授权控制、数据安全、威胁防护与安全运营。

---

## 🎯 安全设计目标

| 目标 | 描述 |
|------|------|
| **零信任架构** | 不信任任何网络边界，所有请求均需认证和授权 |
| **纵深防御** | 多层安全控制，单点失效不影响整体安全 |
| **隐私优先** | 默认本地处理，数据最小化采集，差分隐私保护 |
| **安全内建** | 安全要求贯穿需求、设计、开发、测试、运维全生命周期 |
| **合规驱动** | 满足 ISO 10218/13482、GDPR/CCPA/PIPL、EU AI Act |
| **可审计** | 所有操作可追溯，关键操作不可篡改 |

| 协议 | 使用场景 | 密钥类型 | Token 有效期 |

| **OAuth 2.0 + PKCE** | 第三方应用 | 非对称 (RS256) | Access: 1h |

| **FIDO2/WebAuthn** | 无密码认证 | 非对称 | 按断言 |

## 🔒 数据安全

### 数据加密层级

| 层级 | 场景 | 算法 | 密钥管理 |
|:--:|------|------|----------|
| **传输加密** | 端→云 / 云内部 / Web | TLS 1.3 (仅 PFS 密码套件) | 自动轮换证书 |
| **传输加密** | 机器人间 | mTLS 1.3 (X.509 ECDSA) | HSM 签发 |
| **静态加密** | PostgreSQL | AES-256-GCM (每行独立密钥) | Keychain/HSM |
| **静态加密** | MinIO 对象存储 | SSE-S3 (AES-256) | 自动密钥管理 |
| **静态加密** | 端侧存储 | AES-256-GCM (设备密钥派生) | TPM 密封 |
| **端到端加密** | 用户间消息 | X3DH + Double Ratchet | 每会话独立密钥 |
| **内存加密** | 敏感数据在内存 | 加密内存页 | TEE / SGX |

### 端到端加密 (E2EE)

```
┌──────────────────────────────────────────────────────────────┐
│                  E2EE 消息加密流程                              │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  Alice 设备                           Bob 设备                │
│  ┌──────────┐                        ┌──────────┐            │
│  │ 身份密钥  │──── 预密钥包 ─────────▶│ 身份密钥  │            │
│  │ 签名预密钥│                        │ 签名预密钥│            │
│  │ 一次性预密钥│                       │ 一次性预密钥│           │
│  └──────────┘                        └──────────┘            │
│                                                              │
│  1. X3DH 密钥协商:                                            │
│     Alice → 获取 Bob 预密钥包 → 计算共享密钥 SK                │
│                                                              │
│  2. Double Ratchet 消息加密:                                   │
│     每条消息使用新的消息密钥 (前向安全)                          │
│     DH Ratchet + Symmetric Ratchet                            │
│                                                              │
│  3. 消息格式:                                                  │
│     header(ciphertext, sender_ratchet_pub, msg_num, prev_chain)│
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 隐私保护三层架构

```
数据产生 (端侧)
    │
    ├── L0: 本地处理 (原始传感器数据)
    │   · 摄像头原始帧 → 本地推理 → 仅输出特征向量
    │   · 麦克风音频 → 本地 ASR → 仅上传文本
    │   · 绝不离开设备
    │
    ├── L1: 去标识化上传 (伪匿名)
    │   · 人脸特征向量 (不可逆)
    │   · 环境语义标签 (无个人数据)
    │   · 聚合统计 (k-匿名 ≥ 5)
    │
    └── L2: 加密云端 (用户可控)
        · 用户明确授权的数据
        · E2E 加密
        · 用户可随时删除/导出
```

## 🛡️ 威胁防护

### 威胁模型

| 威胁 | 风险等级 | 防护措施 |
|------|:--:|----------|
| **未授权访问** | 🔴 高 | JWT + OAuth + RBAC + mTLS |
| **数据泄露** | 🔴 高 | E2EE + 静态加密 + 数据最小化 |
| **DDoS 攻击** | 🟡 中 | CDN + WAF + 网关限流 + Sentinel |
| **撞库攻击** | 🟡 中 | 3层防御 (IP限制 + CAPTCHA + 延时) |
| **中间人攻击** | 🔴 高 | TLS 1.3 + 证书固定 (Certificate Pinning) |
| **固件篡改** | 🔴 高 | 安全启动链 + ECDSA 签名验证 |
| **恶意技能** | 🟡 中 | 沙箱隔离 + 权限管控 + 代码扫描 |
| **物理攻击** | 🟢 低 | 防篡改外壳 + 安全 MCU + 自毁逻辑 |
| **供应链攻击** | 🟡 中 | 依赖扫描 + SBOM + 签名验证 |
| **模型投毒** | 🟡 中 | 联邦学习差分隐私 + 模型签名 |

### 异常检测模型

```
异常评分 = Σ wᵢ × scoreᵢ

维度:
  w₁=0.25  地理位置异常 (新IP/新国家/不可能旅行)
  w₂=0.20  登录时间异常 (非活跃时段)
  w₃=0.20  请求频率异常 (突增/爬虫模式)
  w₄=0.15  设备指纹变化 (浏览器/OS/分辨率)
  w₅=0.10  操作序列异常 (偏离正常模式)
  w₆=0.10  数据访问量异常 (批量下载/拖库)

评分 > 0.7 → 触发 MFA 验证
评分 > 0.85 → 会话失效 + 告警
评分 > 0.95 → 账户冻结 + 安全事件
```

### 安全事件响应 SLA

| 严重级别 | 定义 | 响应时间 | 修复时间 |
|:--:|------|:--:|:--:|
| **P0 严重** | 远程代码执行、认证绕过、大规模数据泄露 | 15 min | 4 h |
| **P1 高危** | 权限提升、敏感信息泄露、会话劫持 | 1 h | 24 h |
| **P2 中危** | 信息泄露 (非敏感)、CSRF、XSS | 4 h | 7 d |
| **P3 低危** | 配置问题、弱密码策略 | 24 h | 30 d |
