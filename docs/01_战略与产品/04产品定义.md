## 📱 QooBotOS (简称 QOS)

**QOS** 是双足人形机器人的操作系统，定位为**开源版 Android**：

> Android = Linux Kernel + HAL + ART + Framework + System Apps + Play Store + Google Services
>
> QooBot  = QOS Kernel + QOS HAL + qoobrain 运行时 + QOS Framework + 出厂技能 + QooStore + qoocloud

### Android → QOS 架构对标

| Android 层 | Android 组件 | QOS 层 | QOS 组件 | 说明 |
|-------------|-------------|--------|----------|------|
| **Kernel** | Linux Kernel (进程/内存/驱动/电源/安全) | **QOS Kernel** | Linux + RT-Preempt 实时补丁 + ROS 2 中间件 | 机器人需要硬实时 |
| **HAL** | 硬件抽象层 (Camera/Audio/Sensor HAL) | **QOS HAL** | qoobody HAL (电机/灵巧手/雷达/IMU/摄像头统一接口) | 硬件厂商接入标准 |
| **Native** | Native Libraries (libc/sqlite/OpenGL) | **QOS Native** | qoocore 推理引擎 + MuJoCo 仿真 + 运动控制库 | 端侧 AI 加速 |
| **Runtime** | ART (Dalvik → ART) | **QOS Runtime** | qoobrain Runtime (Python AI 引擎 + C++ 实时引擎) | 双引擎：AI 推理 + 实时控制 |
| **Framework** | Java API Framework (60+ System Services) | **QOS Framework** | QOS System Server + 技能 SDK API | 机器人系统服务 |
| **System Apps** | Launcher/Phone/Settings... | **出厂技能** | 视觉/导航/语音/安全/遥控... | 开机即用 |
| **App Store** | Google Play | **QooStore** | qoostore 技能市场 | 第三方技能分发 |
| **Cloud** | Google Services (GMS) | **qoocloud** | 远程推理/OTA/设备管理/数字孪生 | 云端能力 |
| **Identity** | Google Account | **QooIDaaS** | qooauth 统一身份 | 设备认证/用户授权 |
| **Developer** | Android Studio + SDK | **qoodev** | CLI + IDE 插件 + 仿真 + 调试 | 开发者工具链 |
| **Security** | SELinux + Keystore + Verified Boot | **QOS Security** | qoocompliance 合规 + SELinux + 安全启动链 | 安全基线 |
| **Supply Chain** | — (各家自建) | **qoochain** | BOM 公开 + 产线设计 + 标定指南 | 第三方制造 |
| **Accessories** | — (MFi 类似) | **qoogear** | MFQ 认证 | 配件生态 |
| **Community** | AOSP 社区 | **qoocommunity** | 文档/贡献/活动/教育 | 开源社区 |

## 🧠 QOS 核心子系统

QOS 由两大核心子系统构成，参考 Android 的"前台交互 + 后台服务"双轨设计：

```
QOS (QooBot OS)
│
├── 🧠 qoobrain — 自主意识大脑 (对标 Android Framework + Runtime)
│   │
│   │  感知 → 认知 → 决策 → 规划 → 控制 → 安全
│   │
│   │  端侧自主运行：机器人独立完成环境感知、任务决策、
│   │  运动规划、实时控制、安全监护，无需网络连接。
│   │
│   └── 技术栈：Python 3.11 (AI 引擎) + C++17 (实时引擎)
│       + TypeScript (Web 前端) + ROS 2 (中间件)
│
└── 🎮 qooremote — 远程人工遥控系统 (对标 Android 辅助功能 + 远程管理)
    │
    │  人在回路：当自主大脑无法应对的长尾场景，
    │  人类操作员可远程接管机器人的感知、决策与控制。
    │
    ├── 遥操作 (动作捕捉 / VR / 手柄实时操控)
    ├── 远程接管 (自主 → 人工无缝切换)
    ├── 远程查看 (视频/传感器画面实时回传)
    ├── 远程对讲 (双向语音通信)
    └── 示教录制 (操作数据采集 → 训练自主模型)
```

### 为什么需要双系统？

| 场景 | 自主大脑 (qoobrain) | 远程遥控 (qooremote) |
|------|---------------------|----------------------|
| 日常家庭服务 | ✅ 端侧自主完成 | — |
| 固定产线巡检 | ✅ 自主导航+检测 | — |
| 陌生复杂环境 | ⚠️ 可能失败 | ✅ 人工接管探索 |
| 精细操作任务 | ⚠️ 成功率不足 | ✅ 遥操作精准控制 |
| 紧急异常情况 | ⚠️ 自主恢复可能不足 | ✅ 人工紧急干预 |
| 数据采集/训练 | — | ✅ 操作数据录制 |
| 安全合规要求 | — | ✅ 人工接管通道 |

> 现代人形机器人无一例外采用"自主 + 人在回路"双模架构。Tesla Optimus、Figure AI、1X NEO、Sanctuary AI 都保留了远程遥控通道。这不是技术不足的妥协，而是**安全监管要求**与**长尾场景兜底**的必然设计。

## 🔌 QOS Binder — 进程间通信总线

参考 Android Binder 设计，QOS 需要自己的 IPC 总线：

| Android Binder 特性 | QOS Binder 实现 |
|---------------------|-----------------|
| **驱动层** | `/dev/qoos-binder` 字符设备驱动 |
| **服务注册** | ServiceManager → QOS ServiceManager (qoos-servicemanager) |
| **AIDL 接口定义** | `.qoosidl` 接口定义语言 → 生成 Python/C++ 桩代码 |
| **同步/异步调用** | 同步 (实时控制) + 异步 (AI 推理结果) |
| **权限检查** | UID/PID 检查 + SELinux 上下文 |
| **死亡通知** | 服务端进程崩溃 → 客户端自动感知 (Binder Death Recipient) |
| **零拷贝** | 传感器数据 (点云/图像) → 共享内存直通，不经过 Binder |
| **实时优先级** | 安全/控制消息 → 高优先级，日志/监控 → 低优先级 |

## 🎮 qooremote — 远程人工遥控系统详细设计

### 系统定位

qooremote 不是独立子项目，而是跨现有子项目的**架构概念**——将分布在 qoodev、qoosvc、qoocloud 中的遥控能力统一为一个系统视角。

对标 Android：**无障碍服务 (Accessibility Service) + 设备管理 (Device Policy) + 远程桌面**

### 遥控能力矩阵

| 能力 | 实现位置 | Android 对标 | 说明 |
|------|----------|-------------|------|
| **动作捕捉遥操作** | qoodev (遥操作录制) | — | 操作员穿戴动捕设备，实时控制机器人全身运动 |
| **VR 沉浸式遥控** | qoodev + qoosvc | — | VR 头显呈现机器人第一视角画面，手柄控制 |
| **手柄/键盘遥控** | qoosvc | Game Controller API | 基础移动与操作，用于调试与紧急接管 |
| **远程视频回传** | qoosvc + qoocloud | Camera2 + WebRTC | 机器人摄像头画面实时回传至遥控端 |
| **远程语音对讲** | qoosvc (语音交互) | Telecom API | 操作员与机器人周围的人双向语音通信 |
| **远程接管** | qoobrain (安全监护) | Device Policy Manager | 自主 → 人工模式无缝切换，保持运动连续性 |

### 自主 ↔ 遥控 切换机制

```
        ┌──────────────────────┐
        │    QOS 运行模式       │
        └──────────┬───────────┘
                   │
       ┌───────────┼───────────┐
       ▼           ▼           ▼
┌──────────┐ ┌──────────┐ ┌──────────┐
│ 完全自主  │ │ 混合模式  │ │ 完全遥控  │
│          │ │          │ │          │
│ 机器人独立│ │ 自主执行  │ │ 人类完全  │
│ 感知/决策 │ │ 人类监督  │ │ 控制机器人│
│ /执行    │ │ 可随时接管│ │ 运动与操作│
└──────────┘ └──────────┘ └──────────┘
    日常        复杂场景      紧急/训练
```

切换触发条件：
- **自动切换**：碰撞风险/未知物体/任务超时/置信度过低 → 自动请求人工接管
- **人工切换**：操作员通过 qooremote 主动接管/交还控制权
- **安全切换**：安全监护检测到危险 → 立即急停并通知操作员

## 🎯 产品哲学

```
Android = Linux Kernel + HAL + ART + Framework + System Apps + Play Store + GMS

QooBot  = QOS Kernel (Linux+RT) + QOS HAL (qoobody) + qoobrain Runtime
          + QOS Framework (System Server + Binder) + 出厂技能
          + QooStore (qoostore) + qoocloud (GMS 对标)
          + QooIDaaS (qooauth) + 开发者工具 (qoodev)
          + 开放供应链 (qoochain)
```

> **QOS 不是做一个机器人，而是做一套让机器人"活"起来的操作系统。**
>
> Android 让手机从功能机变成智能机——不是因为硬件更强，而是因为有了**应用生态**。
> QOS 同样如此——不是因为电机更快或雷达更准，而是因为机器人可以**安装技能**、**扩展能力**、**持续进化**。
>
> **QooBot 与宇树的本质区别**：宇树是"闭源版 Apple"——电机/灵巧手/激光雷达全部自研但设计不公开，只有宇树能制造。QooBot 是"开源版 Android"——同样全部自研，但设计图纸、BOM、工艺全部公开，**全世界任何人都能基于 QooBot 设计制造自己的人形机器人**。
>
> 自主大脑让机器人拥有"意识"，远程遥控让人类拥有"兜底"——两者缺一不可，共同构成 QOS 的完整操作系统体验。

## 📚 Android 设计模式 → QOS 设计决策

| Android 设计决策 | 为什么 | QOS 借鉴 |
|------------------|--------|----------|
| **Binder IPC** | 相比 Socket/管道，Binder 支持同步异步、权限检查、死亡通知、零拷贝 | QOS Binder — 机器人需要实时 IPC + 安全 + 可靠性 |
| **HAL 层** | 硬件厂商只需实现 .so 接口，Framework 不变 | qoobody HAL — 任何电机/雷达厂商实现 HAL 即可接入 |
| **Zygote 预加载** | 加速应用启动，共享只读内存页 | qoos-zygote — 预加载 AI 模型到共享内存，技能启动 < 1s |
| **init.rc 声明式启动** | 依赖关系清晰，启动顺序可控 | qoos.rc — 声明式定义各服务的启动顺序和依赖 |
| **SELinux 强制访问** | 即使 root 也无法绕过，安全基线 | QOS SELinux — 安全服务运行在强制访问控制下 |
| **权限分级 (normal/dangerous/signature)** | 平衡安全与易用 | QOS 权限模型 — 摄像头访问需用户授权，急停仅系统可用 |
| **PackageManager** | 统一管理应用的安装/更新/卸载/权限 | SkillManager — 统一管理技能的安装/更新/卸载/权限 |
| **OTA Seamless (A/B 分区)** | 升级失败自动回滚，不影响使用 | QOS OTA — A/B 分区，升级失败自动回滚到上一个版本 |
| **Treble (Framework/ Vendor 分离)** | Framework 升级不依赖 Vendor 实现 | QOS Treble — qoobrain 升级不依赖具体硬件厂商 HAL 实现 |
