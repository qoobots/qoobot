## 📐 平台定义

QooBot 覆盖 7 个运行时平台。**整体收敛原则**：前端全域统一到 TypeScript + Vue3，web_portal/mobile_app/mini_program/mobile_h5 复用同一套语法、组件、接口逻辑，仅通过编译工具输出不同端产物；后端保持 Java + Spring Boot 单一技术栈；PC 专业调试端保留 Python 栈以原生适配 ROS2；机器人端侧为硬件控制专属刚需栈，通过统一通信协议降低跨端对接复杂度。

### 各平台最终技术栈（最大化复用、最小化新增栈）

| # | 平台 | 技术栈 | 补充配套 | 说明 |
|---|------|--------|----------|------|
| 1 | **cloud_service** | Java + Spring Boot 微服务 + PostgreSQL | Redis（缓存/会话管理）、Protobuf（全平台统一消息格式） | 全平台用户管理/设备接入/数据存储/业务逻辑中枢，技术栈保持稳定，不新增编程语言 |
| 2 | **desktop_console** | Python 3.10+ + PySide6 | rclpy（ROS2 原生对接）、PyQtGraph（实时数据曲线）、OpenCV（视觉流渲染） | 面向研发调试场景，必须原生对接 ROS2，Python 栈开发效率最高，无法被前端栈替代，保留独立技术栈，仅用于专业调试端 |
| 3 | **web_portal** | TypeScript + Vue3 + Element Plus + Vite + Electron | — | 作为前端主栈基准，所有业务组件、API 封装、状态管理逻辑均以此为标准，向 H5、小程序、App 端复用；Electron 用于打包 PC 桌面轻量运维端，和网页端共用代码 |
| 4 | **mobile_app** | TypeScript + Vue3 + UniApp | uView Plus（移动端 UI）、uni-socket（WebSocket 实时通信） | 完全沿用 PC 端的 TS+Vue3 语法体系，业务逻辑、接口请求、数据结构 100% 复用 PC 网页端代码；同一工程可直接编译出 Android/iOS 安装包，无需引入 Flutter/React Native 等全新技术栈 |
| 5 | **mini_program** | TypeScript + Vue3 + UniApp | — | 与手机 App 共用同一套 UniApp 工程，直接编译输出微信/支付宝小程序；UI 组件、通信逻辑、业务层代码完全复用，仅需适配小程序端的权限、蓝牙等原生能力 |
| 6 | **mobile_h5** | TypeScript + Vue3 + UniApp | — | 从 UniApp 工程直接编译出 H5 产物，和 PC 网页端共享 API、状态管理、工具函数，几乎零额外开发成本；也可内嵌至 App、小程序 WebView 中作为通用页面 |
| 7 | **onboard_core** | 分层技术栈（硬件控制刚需，无法替代） | 运动控制核心层：C++17 + ROS 2 Jazzy (rclcpp) + Eigen + Pinocchio；功能应用层：C++/Python；底层固件层：C + FreeRTOS + micro-ROS；系统基座：Ubuntu 22.04 + PREEMPT_RT 实时内核 | 内部统一采用 ROS2 DDS 通信，对外通过 rosbridge_suite 提供标准 WebSocket 接口，消息格式与云端、前端共用 Protobuf 定义 |

## 🔌 QOS Binder — 进程间通信总线

> 对标 Android Binder，运行在 `onboard_core` 平台。

| Android Binder 特性 | QOS Binder 实现 | 平台 |
|---------------------|-----------------|------|
| **驱动层** | `/dev/qoos-binder` 字符设备驱动 | onboard_core |
| **服务注册** | qoos-servicemanager (对标 ServiceManager) | onboard_core |
| **AIDL 接口定义** | `.qoosidl` → 生成 Python/C++ 桩代码 | onboard_core |
| **同步/异步调用** | 同步(实时控制) + 异步(AI 推理结果) | onboard_core |
| **权限检查** | UID/PID + SELinux 上下文 | onboard_core |
| **死亡通知** | Binder Death Recipient → 服务崩溃自动感知 | onboard_core |
| **零拷贝** | 传感器数据(点云/图像) → 共享内存直通 | onboard_core |
| **实时优先级** | 安全/控制→REALTIME，日志/监控→LOW | onboard_core |

## 🧠 QOS 核心子系统（跨平台视角）

QOS 由两大核心子系统构成：

```
QOS (QooBot OS)
│
├── 🧠 qoobrain — 自主意识大脑
│   │  平台：onboard_core (主) + cloud_service (推理增强) + web_portal (状态面板)
│   │
│   │  感知 → 认知 → 决策 → 规划 → 控制 → 安全
│   │  端侧自主运行，无需网络连接
│   │  技术栈：Python 3.11 + C++17 + TypeScript + ROS 2
│   │
│   └── 对标：Android Framework + Runtime
│
└── 🎮 qooremote — 远程人工遥控系统
    │  平台：onboard_core + cloud_service + desktop_console + web_portal + mobile_app
    │
    │  人在回路：当自主大脑无法应对长尾场景时，
    │  人类操作员可远程接管机器人的感知、决策与控制。
    │
    ├── 遥操作 (动作捕捉/VR/手柄)     → desktop_console + onboard_core
    ├── 远程接管 (自主↔人工无缝切换)   → onboard_core + cloud_service
    ├── 远程查看 (视频/传感器回传)     → onboard_core + cloud_service + web_portal/mobile
    ├── 远程对讲 (双向语音通信)        → onboard_core + cloud_service + mobile_app
    └── 示教录制 (操作数据采集→训练)   → desktop_console + cloud_service
```

### 自主 ↔ 遥控 切换机制

```
        ┌──────────────────────┐
        │    QOS 运行模式       │  平台：onboard_core
        └──────────┬───────────┘
                   │
       ┌───────────┼───────────┐
       ▼           ▼           ▼
┌──────────┐ ┌──────────┐ ┌──────────┐
│ 完全自主  │ │ 混合模式  │ │ 完全遥控  │
│ 机器人独立│ │ 自主执行  │ │ 人类完全  │
│ 感知/决策 │ │ 人类监督  │ │ 控制机器人│
│ /执行    │ │ 可随时接管│ │ 运动与操作│
└──────────┘ └──────────┘ └──────────┘
    日常        复杂场景      紧急/训练
```

切换触发条件：
- **自动切换**：碰撞风险/未知物体/任务超时/置信度过低 → 自动请求人工接管 (onboard_core → cloud_service → mobile_app 告警)
- **人工切换**：操作员通过 desktop_console/web_portal/mobile_app 主动接管/交还控制权
- **安全切换**：安全监护检测到危险 → 立即急停并通知操作员 (onboard_core → cloud_service → 全平台推送)

| Android 设计决策 | 为什么 | QOS 借鉴 | 涉及平台 |
|------------------|--------|----------|----------|

| **init.rc 声明式启动** | 依赖关系清晰，启动顺序可控 | qoos.rc — 声明式定义服务启动顺序和依赖 | onboard_core |
| **SELinux 强制访问** | 即使 root 也无法绕过 | QOS SELinux — 安全服务在强制访问控制下 | onboard_core |
| **权限分级** | normal/dangerous/signature 三级 | QOS 权限模型 — dangerous 需 web/mobile 确认 | onboard_core + web/mobile |

| **Google Play** | 统一应用分发 | QooStore — 技能发现/分发/商业化 | cloud + web + mobile |

```
                     qoocloud / qoostore / qooauth / qoochain / qoocompliance / qoogear / qoocommunity

                     qoodev (CLI + IDE + 仿真 + 遥操作录制) — 独立栈，ROS2 刚需
  web_portal       ← TS + Vue3 + Element Plus + Vite + Electron
                     qooweb / qoocloud控制台 / qoostore商店 / 管理后台 — 前端主栈基准
  mobile_app       ← TS + Vue3 + UniApp + uView Plus（100% 复用 web_portal 业务代码）
                     qoostore / qoocloud / qooauth / qoocommunity
  mini_program     ← TS + Vue3 + UniApp（与 mobile_app 共用工程）
                     qoostore / qoocloud / qooauth / qoocommunity（轻量）
  mobile_h5        ← TS + Vue3 + UniApp（从 UniApp 编译 H5 产物，零额外开发成本）
                     qooweb / qoostore / qoocloud / qooauth / qoocommunity（响应式）

全域统一通信：
  · 业务接口 → RESTful 规范
  · 实时数据 → WebSocket 协议
  · 消息格式 → Protobuf 定义，自动生成 Java/TS/C++/Python 多端代码
  · 机器人内部 → ROS2 DDS，对外通过 rosbridge_suite 暴露 WebSocket
```

> **QOS 不是做一个机器人，而是做一套让机器人"活"起来的操作系统。**
>
> Android 让手机从功能机变成智能机——不是因为硬件更强，而是因为有了**应用生态**。
> QOS 同样如此——不是因为电机更快或雷达更准，而是因为机器人可以**安装技能**、**扩展能力**、**持续进化**。
>
> **QooBot 与宇树的本质区别**：宇树是"闭源版 Apple"——电机/灵巧手/激光雷达全部自研但设计不公开，只有宇树能制造。QooBot 是"开源版 Android"——同样全部自研，但设计图纸、BOM、工艺全部公开，**全世界任何人都能基于 QooBot 设计制造自己的人形机器人**。
>
> 自主大脑(qoobrain)让机器人拥有"意识"，远程遥控(qooremote)让人类拥有"兜底"——两者缺一不可，共同构成 QOS 的完整操作系统体验。
>
> 7 个平台覆盖确保用户在任何场景都能与机器人交互——PC 端开发调试、Web 端深度管理、手机 App 日常监控、小程序轻量触达、H5 零安装访问。
>
> **技术栈收敛哲学**：前端全域统一到 TypeScript + Vue3，web_portal/mobile_app/mini_program/mobile_h5 共享同一套语法、组件、接口逻辑，仅通过编译工具输出不同端产物，最大程度降低前端团队的学习与维护成本。后端统一 Java + Spring Boot，不新增编程语言。PC 调试端保留 Python 栈（ROS2 刚需），onboard_core 保留 C++/C/Python 分层栈（硬件控制刚需）。全域统一 Protobuf 消息格式 + RESTful/WebSocket 协议，一处定义，多端自动生成，消除跨端对接成本。
