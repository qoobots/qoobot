# QooBot 组织架构设计

> 最后更新：2026-05-29
>
> 本文档定义 QooBot 双足仿生人项目的组织架构、团队分工、治理模式与协作机制。

---

## 1. 组织定位

QooBot 采用**开源社区 + 商业实体**双轮驱动模式：

```
┌─────────────────────────────────────────────────────────┐
│                    QooBot 组织架构                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   ┌──────────────────┐    ┌──────────────────────┐      │
│   │   QooBot 基金会    │    │   QooBot Technologies  │      │
│   │   (非营利开源治理)   │    │   (商业运营实体)        │      │
│   └────────┬─────────┘    └──────────┬───────────┘      │
│            │                         │                   │
│            └──────────┬──────────────┘                   │
│                       ▼                                  │
│            ┌─────────────────────┐                       │
│            │    开源社区贡献者      │                       │
│            │   (全球开发者/企业)    │                       │
│            └─────────────────────┘                       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

- **QooBot 基金会**：负责开源项目治理、技术方向决策、社区运营、知识产权管理
- **QooBot Technologies**：负责商业产品开发、生产制造、市场销售、客户服务

## 2. 核心团队架构

### 2.1 顶层组织

```
                         ┌─────────────┐
                         │  技术委员会    │
                         │  (TSC)       │
                         └──────┬──────┘
                                │
        ┌───────────────────────┼───────────────────────┐
        │                       │                       │
        ▼                       ▼                       ▼
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  软件平台部    │    │  硬件平台部    │    │  生态与社区部  │
│  Software     │    │  Hardware     │    │  Ecosystem    │
└──────────────┘    └──────────────┘    └──────────────┘
```

### 2.2 部门与子项目映射

| 部门 | 职责 | 对应子项目 |
|------|------|------------|
| **软件平台部** | 机器人操作系统、中间件、应用框架 | qoobrain、qoocore、qooweb、qooauth |
| **硬件平台部** | 机器人本体设计、电子设计、生产制造 | qoobody、qoogear |
| **生态与社区部** | 开发者生态、技能市场、社区运营 | qoostore、qoocommunity、qoodev |
| **云端服务部** | 云平台、数据服务、远程运维 | qoocloud、qoosvc、qooremote |
| **合规与安全部** | 安全认证、合规审查、质量保障 | qoocompliance、qoochain |

## 3. 技术委员会（TSC）

### 3.1 职责

| 职责 | 说明 |
|------|------|
| **技术方向决策** | 制定技术路线图、架构评审、技术选型 |
| **子项目审批** | 新子项目孵化审批、废弃决策 |
| **SIG 管理** | 设立/解散 Special Interest Group |
| **版本发布审批** | 重大版本发布审批、API 兼容性决策 |
| **争议仲裁** | 技术争议最终裁决 |

### 3.2 组成

| 角色 | 人数 | 产生方式 |
|------|------|----------|
| **主席** | 1 | 基金会任命 |
| **核心委员** | 5-7 | 核心 Maintainer 选举 |
| **轮值委员** | 2-3 | 社区贡献者选举，任期 1 年 |
| **秘书** | 1 | 基金会委派 |

### 3.3 决策机制

| 类型 | 表决方式 | 通过门槛 |
|------|----------|----------|
| 日常决策 | 线上投票（Lazy Consensus） | 72h 无反对即通过 |
| 技术选型 | TSC 会议投票 | 2/3 多数 |
| 架构变更 | TSC 会议投票 + 社区 RFC | 2/3 多数 + RFC 公示 2 周 |
| 子项目孵化/废弃 | TSC 投票 | 2/3 多数 |

## 4. SIG（Special Interest Group）体系

### 4.1 SIG 列表

| SIG 名称 | 领域 | 组长 | 状态 |
|----------|------|------|------|
| **SIG-Mechanical** | 机械结构设计 | qoobody/mechanical Maintainer | 活跃 |
| **SIG-Electronics** | 电子电路设计 | qoobody/electronics Maintainer | 活跃 |
| **SIG-Motion** | 运动规划与控制 | qoobrain 运动模块 Maintainer | 活跃 |
| **SIG-Perception** | 感知与视觉 | qoobrain 感知模块 Maintainer | 活跃 |
| **SIG-AI** | 端侧 AI 与模型 | qoocore AI 模块 Maintainer | 活跃 |
| **SIG-Safety** | 功能安全 | qoocompliance Maintainer | 规划中 |
| **SIG-Cloud** | 云端服务 | qoocloud Maintainer | 活跃 |
| **SIG-Web** | Web 前端与门户 | qooweb Maintainer | 活跃 |
| **SIG-Mobile** | 移动端与小程序 | mobile_app Maintainer | 活跃 |
| **SIG-Docs** | 文档与教程 | 文档负责人 | 活跃 |
| **SIG-DevTools** | 开发工具与仿真 | qoodev Maintainer | 活跃 |

### 4.2 SIG 运作规范

| 事项 | 要求 |
|------|------|
| 会议频率 | 双周例会 |
| 决策机制 | Lazy Consensus + 组长裁决 |
| 新成员加入 | PR 贡献 ≥ 3 次 → 申请 → 组长审批 |
| 文档产出 | 每次会议纪要公开 |
| 代码审查 | 至少 1 位 Maintainer + 1 位 Committer 批准 |

## 5. 角色与权限体系

### 5.1 技术角色

| 角色 | 权限 | 晋升条件 |
|------|------|----------|
| **User** | 使用项目、提 Issue | 注册 GitHub |
| **Contributor** | 提交 PR、参与讨论 | 至少 1 个 PR 被合并 |
| **Committer** | 代码审查、合并 PR | 持续贡献 3 个月 + 10+ 合并 PR + 现有 Committer 推荐 |
| **Maintainer** | 子项目决策、发布管理 | Committer 6 个月 + 技术委员会批准 |
| **TSC Member** | 技术方向决策 | 选举或任命 |

### 5.2 运营角色

| 角色 | 职责 | 人数 |
|------|------|------|
| **社区经理** | 社区运营、活动组织、内容策划 | 2-3 |
| **开发者关系** | 开发者体验、文档、教程 | 2-3 |
| **布道师** | 技术演讲、博客、社交媒体 | 2-3 |
| **翻译协调员** | 多语言文档管理 | 按语种 |
| **活动策划** | Hackathon、开发者大会 | 1-2 |

## 6. 协作流程

### 6.1 代码贡献流程

```
Issue 创建 → 讨论确认 → Fork → 开发 → PR 提交 → CI 检查 → Code Review → 合并
```

| 环节 | 时间预期 | 负责人 |
|------|----------|--------|
| Issue 响应 | 48h 内 | Maintainer |
| PR Review | 3 个工作日内 | Committer/Maintainer |
| CI 检查 | 自动触发 | CI Bot |
| 合并决策 | Review 通过后 | Maintainer |

### 6.2 设计贡献流程

```
RFC 提案 → 社区讨论(2周) → TSC 评审 → 设计实施 → PR → Review → 合并
```

### 6.3 发布流程

```
功能冻结 → 分支创建 → 测试验证 → RC 发布 → 社区测试(2周) → 正式发布
```

## 7. 激励机制

| 激励类型 | 对象 | 方式 |
|----------|------|------|
| **社区声望** | 所有贡献者 | GitHub 贡献墙、社区排行榜、Contributor 徽章 |
| **物资激励** | 活跃贡献者 | 开发套件折扣、周边礼品、大会门票 |
| **经济激励** | 核心贡献者 | qoostore 技能分成、Bug Bounty、GSoC/OSPP 奖金 |
| **职业发展** | 核心成员 | 认证证书、就业推荐、演讲机会 |
| **共治权利** | Maintainer+ | TSC 投票权、子项目方向决策 |
