# qoocompliance — 安全架构设计

> 版本：v0.1 | 最后更新：2026-06-29 | 状态：Draft | 子项目：qoocompliance（法规合规）

---

## 1. 安全设计原则

1. **合规即安全**：合规框架天然包含安全要求，两者一体化设计
2. **审计不可抵赖**：所有合规操作留痕，不可篡改
3. **最小权限**：合规平台按角色严格授权
4. **证据完整性**：合规证据哈希上链（qoochain 可选）防篡改

---

## 2. 访问控制

### 2.1 角色定义

| 角色 | 权限 |
|:-----|------|
| Compliance Admin | 全部权限：创建/编辑/审批/删除 |
| Compliance Reviewer | 查看清单、更新检查项状态、添加证据 |
| Engineer | 查看分配给自己的清单项、上传证据 |
| Auditor (只读) | 查看所有数据、导出审计报告 |
| Project Manager | 查看进度、导出报告、不能修改 |

## 3. 审计追踪

### 3.1 审计日志结构

```json
{
  "audit_id": "AUD-20260629-001234",
  "timestamp": "2026-06-29T10:15:00Z",
  "user_id": "user_zhang",
  "user_role": "engineer",
  "action": "checklist_item.update",
  "entity_type": "checklist_item",
  "entity_id": "CL-EU-MD-001",
  "changes": {
    "status": {"from": "pending", "to": "passed"},
    "evidence": {"from": [], "to": ["safety_review_v3.pdf"]}
  },
  "ip_address": "10.0.1.45",
  "user_agent": "Mozilla/5.0 ..."
}
```

### 3.2 审计保证

- 审计日志写入 WAL (Write-Ahead Log) 确保持久化
- 定期计算审计日志 Merkle Tree 根哈希
- 可选：将根哈希提交到区块链（qoochain）防篡改
- 审计日志保留期 ≥ 法规要求（GDPR: 3年, 医疗: 10年）

## 4. 数据保护

### 4.1 数据分类

| 级别 | 数据 | 保护措施 |
|:----:|------|---------|
| L1 | 产品设计文件、测试报告 | 加密存储 + 访问控制 |
| L2 | 检查清单、合规决策 | 加密存储 + 审计日志 |
| L3 | 法规文本（公开） | 完整性校验 |
| L4 | 审计日志 | 不可删除 + 哈希链 |

### 4.2 合规证据管理

- 证据文件存储在 MinIO (S3 兼容)，服务端加密
- 上传时计算 SHA-256 哈希并记录
- 证据文件不可被覆盖，仅追加新版本
- 证据链完整性可验证

## 5. 安全合规自身

qoocompliance 本身也需要满足安全合规要求：

| 要求 | 措施 | 状态 |
|:-----|------|:----:|
| 访问控制 | RBAC + qooauth SSO | ✅ |
| 审计日志 | 全操作审计 + 哈希链 | ✅ |
| 数据加密 | AES-256 存储 + TLS 传输 | ✅ |
| 漏洞管理 | 定期安全扫描 | 🚧 |
| 备份恢复 | 每日备份 + 跨区域 | 🚧 |
| 业务连续性 | BCP 计划 | 🚧 |
