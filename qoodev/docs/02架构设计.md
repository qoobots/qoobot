# qoodev 架构设计

> 最后更新：2026-06-27
>
> **设计原则**：模块化、可扩展、高性能、开发者友好。qoodev 不是单体 IDE，而是一个松耦合的工具链平台，各模块既可独立使用，也可通过统一总线协同工作。

## 2. 分层说明

### 2.1 用户界面层

提供多种交互入口，覆盖不同开发者场景：

| 入口 | 适用场景 | 技术方案 |
|------|---------|----------|
| **VS Code Plugin** | 主力开发环境，Python/C++ 混合开发 | TypeScript + LSP |
| **JetBrains Plugin** | C++ 重度开发、行为树可视化编辑 | Kotlin/Java + LSP |
| **Web Dashboard** | 仿真监控、性能面板、远程调试 | React + Three.js |
| **CLI (`qoo`)** | 项目脚手架、CI/CD 集成、批量操作 | Python/Rust |

### 2.2 核心服务层

独立于 UI 的后台服务进程，通过 Message Bus 与 UI 层和引擎层通信：

| 服务 | 职责 | 通信协议 |
|------|------|---------|
| **LSP Server** | 语法高亮、智能补全、代码导航、诊断 | LSP (JSON-RPC) |
| **Debug Adapter** | 断点管理、变量监控、调用栈追踪 | DAP (Debug Adapter Protocol) |
| **Project Manager** | 项目模板、依赖解析、构建编排、打包签名 | gRPC |

### 2.3 工具引擎层

各模块的核心计算引擎，负责繁重工作：

| 引擎 | 核心能力 |
|------|---------|
| **Simulation Engine** | 物理仿真生命周期管理、场景加载、多引擎适配 |
| **Debugger Engine** | 运行时状态快照、回放录制/播放、行为树状态追踪 |
| **Profiler Engine** | 全链路延迟采集、资源火焰图生成、通信带宽统计 |
| **Data Labeler Engine** | 2D/3D 标注渲染、标注格式转换、质量校验 |

### 2.4 运行时/桥接层

与 qoobrain 运行时的通信层：

| 桥接模块 | 说明 |
|----------|------|
| **qoobrain API Client** | 封装 qoobrain 的所有调试/监控 API |
| **ROS 2 Bridge** | DDS 消息订阅/发布，用于传感器数据流和控制指令 |
| **Sim Bridge** | 统一仿真器接口（Isaac Sim / MuJoCo / Gazebo） |

## 3. 模块间通信

```
                  ┌──────────────────────┐
                  │    Message Bus        │
                  │  (gRPC + WebSocket)   │
                  └──┬───┬───┬───┬───────┘
                     │   │   │   │
        ┌────────────┘   │   │   └────────────┐
        ▼                ▼   ▼                ▼
   ┌─────────┐    ┌─────────┐  ┌─────────┐  ┌─────────┐
   │  IDE    │    │Debugger │  │Profiler │  │ Project │
   │ Plugin  │    │ Engine  │  │ Engine  │  │ Manager │
   └─────────┘    └─────────┘  └─────────┘  └─────────┘
```

- **gRPC**：核心服务间高性能调用（控制指令、状态查询）
- **WebSocket**：实时数据推送（传感器流、日志流、性能指标）
- **DDS (ROS 2)**：与机器人运行时的传感器/控制数据交换
- **Shared Memory**：同机大数据传输（图像流、点云流）

## 4. 数据流

### 4.1 开发调试流程

```
开发者编写代码
     │
     ▼
IDE Plugin (语法检查/补全)
     │
     ▼
qoo build (编译打包)
     │
     ▼
Simulation Engine (加载仿真场景)
     │
     ▼
qoobrain Runtime (运行技能)
     │
     ├──► Debugger Engine (断点/变量/行为树状态)
     ├──► Profiler Engine (延迟/资源/通信)
     └──► Data Recorder (传感器+控制数据录制)
              │
              ▼
         Data Labeler (标注/分析)
```

### 4.2 传感器数据流

```
机器人/仿真器
     │
     ▼
ROS 2 Bridge / Sim Bridge
     │
     ├──► Sensor Visualizer (Web Dashboard)
     ├──► Data Recorder (存储为 .qoodata)
     └──► Profiler Engine (延迟分析)
```

## 5. 技术栈

| 层次 | 技术选型 | 理由 |
|------|---------|------|
| IDE Plugin | TypeScript (VS Code), Kotlin (JetBrains) | 生态标准 |
| Web Dashboard | React + TypeScript + Three.js | 3D 可视化需求 |
| CLI | Python 3.11+ (主), Rust (性能关键路径) | 开发者熟悉度高 |
| Core Services | Python (主), C++17 (性能关键) | 与 qoobrain 一致 |
| Simulation | Python + C++ | 仿真器 SDK 绑定 |
| Message Bus | gRPC + WebSocket + ZeroMQ | 高性能 + 实时推送 |
| 数据存储 | SQLite (本地), Parquet (数据集), Protobuf (序列化) | 轻量 + 高效 |
| 打包格式 | .qooskills = ZIP + manifest.json + 签名 | 可扩展 + 安全 |

## 7. 安全设计

| 层面 | 措施 |
|------|------|
| **代码签名** | 对接 qooauth 开发者证书，技能包数字签名 |
| **沙箱隔离** | 仿真环境容器化运行，防止恶意技能逃逸 |
| **权限控制** | 技能声明权限清单，运行时权限校验 |
| **数据安全** | 录制的传感器数据本地加密存储 |
| **供应链安全** | 依赖包签名校验，SBOM 生成 |
