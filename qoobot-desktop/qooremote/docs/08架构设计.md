# 08 — 架构设计

> 版本：v0.1.0 | 最后更新：2026-06-30 | 状态：Draft
> 子项目：qooremote（远程机器人监控遥控工具）

---

## 1. 设计目标与原则

### 1.1 核心设计目标

| 目标 | P0 指标 | P1 目标 | 说明 |
|------|---------|---------|------|
| 视频端到端延迟(局域网) | < 200ms | < 100ms | WebRTC P2P 直连 |
| 视频端到端延迟(广域网) | < 500ms | < 300ms | TURN 中继优化 |
| 控制指令延迟 | < 50ms(局域网) / < 100ms(WAN) | < 30ms / < 60ms | DataChannel 低延迟通道 |
| 语音往返延迟 | < 150ms | < 100ms | Opus 低延迟模式 |
| 紧急制动响应 | < 100ms | < 50ms | 独立信令通道 |
| 连接恢复时间 | < 5s | < 3s | 自动重连 + 状态恢复 |
| 桌面端启动时间 | < 3s | < 2s | 懒加载模块 |
| 桌面端内存占用 | < 1GB | < 512MB | 按需加载面板 |

### 1.2 架构原则

| 原则 | 说明 |
|------|------|
| 分层解耦 | UI → ViewModel → Service → Platform 四层分离 |
| 协议优先 | 通信协议设计先于具体实现，确保跨平台一致性 |
| 安全内置 | 全链路 TLS 加密、控制指令签名、紧急制动独立通道 |
| 离线降级 | 连接中断时界面保持可用，提供历史数据查看 |
| 可扩展操控 | 插件化的输入设备支持（手柄/动捕/VR） |
| 跨端一致 | 桌面/Web/移动端核心交互逻辑一致 |

## 3. 数据流架构

### 3.1 控制指令流 (上行)

```
操作员输入                    qooremote                    qoocloud                   Robot
─────────                    ─────────                    ────────                   ─────
手柄/键盘 ──► InputManager ──► TeleopVM ──► DataChannel ──► (中继) ──► qoobrain ──► qoobody
                                   │                                                执行器
                                   │ HMAC 签名
                                   ▼
                              AuditLog
```

### 3.2 状态上报流 (下行)

```
Robot Side                          qoocloud                         qooremote
──────────                          ────────                         ─────────
qoosvc ──► DataChannel ──► (中继) ──► WebSocket ──► RobotVM ──► UI 更新
                                                           │
                                                           ▼
                                                     Dashboard
                                                     VideoPanel
                                                     AlertList
```

### 3.3 音视频流 (P2P 直连)

```
Robot Camera ──► Encoder (H.264) ──► WebRTC ──╮
                                              │ P2P (NAT穿透后)
                                              │  或 TURN 中继
                                              │
Operator Speaker ◄── Decoder (Opus) ◄─ WebRTC ─╯
```

## 4. 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 桌面框架 | PySide6 | Qt 生态成熟，原生性能，跨平台，OpenGL 集成好 |
| Web 框架 | Vue 3 + Element Plus | 与 qooweb 技术栈一致，组件丰富 |
| 移动框架 | UniApp | 一套代码多端发布(iOS/Android/H5) |
| 信令协议 | WebSocket (JSON) | 全平台原生支持，双向低延迟 |
| 媒体协议 | WebRTC | P2P 直连最低延迟，SRTP 安全传输 |
| 录制格式 | HDF5 | 支持多通道异构数据、压缩、随机读取 |
| 3D 渲染 | OpenGL (desktop) / Three.js (web) | 原生性能 vs Web 兼容性 |
| 输入设备 | 插件化接口 | 支持社区贡献手柄/动捕/VR 驱动 |
| 构建系统 | setuptools + Vite + HBuilderX | 各平台标准构建工具 |

## 7. 技术债务与演进方向

| 问题 | 当前状态 | 演进方向 |
|------|----------|----------|
| 桌面端 3D 渲染 | 规划中使用 PyOpenGL | 评估 Qt 3D / OpenGL ES 方案 |
| WebRTC 实现 | 桌面端需引入 Python WebRTC 库 | 评估 aiortc / libwebrtc 绑定 |
| 插件热加载 | 启动时静态加载 | 支持运行时动态加载/卸载插件 |
| 跨端代码复用 | 三套独立代码 | 探索核心逻辑 WASM 共享方案 |
