# 02 — 架构设计

> 版本：v0.4 | 最后更新：2026-06-29 | 状态：Stable
> 子项目：qoocompliance（法规合规）| 架构：Java Spring Boot REST 服务
> 对标：CE/FCC 认证体系 + GDPR 合规框架 + ISO 机器人安全标准族

---

## 1. 设计目标与原则

### 1.1 核心设计目标

| 目标 | P0 指标 | P1 目标 | 说明 |
|------|---------|---------|------|
| 市场准入覆盖 | 中国 + 欧盟 + 北美 | + 日本 + 韩国 + 东南亚 | 目标市场合规检查清单覆盖 |
| 法规更新响应 | < 7 天 | < 24 小时 | 法规变化→差距分析→整改提醒 |
| 合规文档生成 | 自动化 80% | 自动化 95% | 技术文档/测试报告/合规声明 |
| 认证追踪 | 实时仪表盘 | 自动延期提醒 | 各市场认证状态与有效期管理 |
| 安全标准覆盖 | ISO 13482/10218/13849 | + ISO 3691-4/TS 15066 | 机器人安全标准族 |

### 1.2 设计原则

- **合规即设计**：合规需求不是事后补丁，而是架构设计的一部分
- **可审计性**：所有合规决策、测试结果、审查记录均可追溯
- **区域适配**：按目标市场自动生成合规检查清单和差距分析
- **持续合规**：法规变化时自动触发差距分析和整改建议
- **模板驱动**：标准化的技术文档模板，减少重复劳动
- **知识库优先**：法规解读、最佳实践、案例库持续积累

### 2.2 模块职责

| 模块 | 职责 | 输入 | 输出 |
|------|------|------|------|
| ComplianceController | 统一 REST 入口 `/api/v1/compliance`，聚合所有业务 Service | HTTP 请求 | JSON 响应 |
| RegulationController | 法规监控 REST 入口 `/api/v1/regulations` | HTTP 请求 | JSON 响应 |
| ComplianceChecklistService | 按市场/产品生成检查清单、进度统计、差距分析 | 项目名 + 目标市场 | ComplianceProject + 检查项列表 |
| RobotSafetyService | ISO 13482/10218/13849 评估、SIL 功能安全、HAZOP/FMEA | 产品安全参数 | SafetyAssessment 报告 |
| WirelessEmcService | FCC/CE RED/SRRC/MIC 认证评估、EMC 测试 | 无线模块参数 | CertificationReport |
| PrivacyDataService | GDPR/CCPA/PIPL 隐私评估、DPIA、传感器隐私 | 数据处理描述 | PrivacyAssessment 报告 |
| AIEthicsService | EU AI Act 合规、算法透明度、偏见检测 | AI 系统描述 | AIActAssessment 报告 |
| ConsumerSafetyService | CE 机械指令/LVD/UL/儿童安全/产品责任评估 | 产品安全参数 | ConsumerSafetyReport |
| ExportControlService | ECCN 分类、加密申报、实体筛查、制裁审查 | 技术参数 | ECCNReport 等 |
| EnvironmentalService | RoHS/WEEE/REACH/碳足迹/能效评估 | 材料清单/能耗数据 | EnvironmentalReport |
| ComplianceManagementService | 文档模板管理、认证进度追踪、审查记录 | 认证/审查数据 | 管理仪表盘 |
| RegulationMonitorService | 法规列表查询、即将生效变更、变更历史 | 市场代码 | Regulation 列表 |

## 3. 技术选型

| 组件 | 技术选型 | 理由 |
|------|---------|------|
| 核心语言 | Java 17 | 企业级 REST 服务，与 qoobot-cloud 统一技术栈 |
| Web 框架 | Spring Boot 3.4 | 生态成熟、自动配置、Actuator 监控 |
| 构建工具 | Maven 3.x | 依赖管理、多模块构建，与 qoobot-cloud 父 POM 集成 |
| REST API | Spring MVC (@RestController) | 声明式路由、Jackson JSON 序列化 |
| 数据存储 | JPA + PostgreSQL | v0.4 持久化方案，7 Entity + 7 Repository |
| 数据库迁移 | Flyway | 版本化数据库迁移（V1/V2/V3） |
| API 文档 | springdoc-openapi (Swagger) | 自动生成 OpenAPI 3.0 文档 |
| 监控 | Spring Boot Actuator | 健康检查、指标暴露 |
| 序列化 | Jackson (内置) | Spring Boot 默认 JSON 处理 |
| DTO 定义 | Java Record + 内部类 | 简洁不可变数据传输对象 |
| 版本控制 | Git | 所有代码和设计文档版本化 |

## 4. 合规工作流

### 4.1 新产品合规流程

```
[新产品立项] 
     │
     ▼
[确定目标市场] → 检查清单生成器 → 按市场生成合规检查清单
     │
     ▼
[产品设计阶段] → 合规即设计
     │
     ├── 安全标准评估 (HAZOP/FMEA)
     ├── 隐私设计审查 (DPIA)
     ├── 无线方案预审 (频谱/EMC)
     └── AI 伦理边界定义
     │
     ▼
[原型验证阶段] → 差距分析引擎
     │
     ├── 逐项检查 vs 法规要求
     ├── 生成差距报告
     └── 整改建议优先级排序
     │
     ▼
[认证测试阶段] → 认证追踪器
     │
     ├── 送检实验室管理
     ├── 测试报告收集
     └── 认证进度仪表盘
     │
     ▼
[市场准入] → 文档生成器
     │
     ├── CE 技术文件
     ├── FCC 测试报告
     ├── 合规声明 (DoC)
     └── 用户手册安全章节
     │
     ▼
[上市后监控] → 法规监控器
     │
     ├── 法规变化追踪
     ├── 事故报告机制
     └── 认证续期管理
```

### 4.2 法规变更响应流程

```
[法规变更检测] → 法规监控器 (每日爬取)
     │
     ▼
[影响分析] → 差距分析引擎
     │
     ├── 影响哪些产品/市场？
     ├── 影响哪些设计决策？
     └── 合规截止日期？
     │
     ▼
[整改提醒] → 通知相关团队
     │
     ├── 生成整改任务
     ├── 分配责任人
     └── 设置截止日期
     │
     ▼
[整改跟踪] → 认证追踪器
     │
     └── 整改完成 → 更新合规状态
```
