## 2. 关键技术决策

### 2.1 密码哈希：argon2id vs bcrypt

| 维度 | argon2id | bcrypt |
|:-----|:---------|:-------|
| **内存硬化** | ✅ 可配置内存成本 | ❌ 无内存硬化 |
| **抗 GPU/ASIC** | ✅ 高 | ⚠️ 中 |
| **OWASP 推荐** | ✅ 首选 | ⚠️ 次选 |
| **Java 支持** | ✅ Bouncy Castle | ✅ jBCrypt |

**决策**：选用 **argon2id**（m=64MB, t=3, p=4），提供更强的抗暴力破解能力。

### 2.2 JWT 签名：Ed25519 vs RS256

| 维度 | Ed25519 | RS256 |
|:-----|:--------|:------|
| **密钥长度** | 32 字节 | 2048+ 位 |
| **签名速度** | ~50μs | ~1ms |
| **安全性** | 128-bit 等效 | 112-bit 等效 |
| **标准化** | RFC 8037 | RFC 7518 |

**决策**：选用 **Ed25519**，更小的密钥和签名尺寸，更高的验证吞吐量。

### 2.3 服务间通信：gRPC vs REST

| 维度 | gRPC (Protobuf) | REST (JSON) |
|:-----|:----------------|:------------|
| **序列化** | Protobuf 二进制 | JSON 文本 |
| **性能** | 高（~10x REST） | 基准 |
| **类型安全** | ✅ 强类型代码生成 | ❌ 运行时校验 |
| **流式支持** | ✅ 原生双向流 | ⚠️ SSE/WebSocket |

**决策**：内部服务间使用 **gRPC + Protobuf**，对外 API 使用 REST + OAS 3.0。

### 2.4 会话存储：Redis 集中式 vs JWT 无状态

| 维度 | Redis 集中式 | JWT 纯无状态 |
|:-----|:-------------|:------------|
| **即时吊销** | ✅ 删除即可 | ❌ 需黑名单 |
| **会话控制** | ✅ 并发/踢出 | ❌ 困难 |
| **扩展性** | ⚠️ Redis 瓶颈 | ✅ 无状态 |
| **延迟** | 网络往返 | 本地验证 |

**决策**：**混合模式** — JWT 短期 Access Token（15min）+ Redis 集中管理 Refresh Token 和黑名单。

## 3. 核心组件设计

### 3.1 认证过滤器链

```
HTTP Request
     │
     ▼
┌─────────────────┐
│  CorsFilter     │  ← CORS 配置
└────────┬────────┘
         ▼
┌─────────────────┐
│  RateLimitFilter│  ← Token Bucket 限流
└────────┬────────┘
         ▼
┌─────────────────┐
│  SecurityFilter │  ← Spring Security Chain
│  ┌─────────────┐│
│  │ JwtAuthFilter│  ← Bearer Token 提取验证
│  ├─────────────┤│
│  │ OAuth2Filter │  ← OAuth2 端点处理
│  ├─────────────┤│
│  │ ApiKeyFilter │  ← API Key 验证
│  ├─────────────┤│
│  │ BasicAuth    │  ← 客户端凭证认证
│  └─────────────┘│
└────────┬────────┘
         ▼
┌─────────────────┐
│  Controller     │  ← 业务处理
└─────────────────┘
```

### 3.2 Token 生命周期管理

```
Access Token (JWT, Ed25519, 15min):
┌──────────────────────────────────────────┐
│ Header: {"alg":"EdDSA","typ":"JWT"}      │
├──────────────────────────────────────────┤
│ Payload: {                               │
│   "sub": "user_xxx",                     │
│   "iss": "qooauth",                      │
│   "iat": 1719667200,                     │
│   "exp": 1719668100,                     │
│   "scopes": ["read:profile","device:*"], │
│   "jti": "uuid_xxx"                      │
│ }                                        │
├──────────────────────────────────────────┤
│ Signature: Ed25519_Sign(header.payload)  │
└──────────────────────────────────────────┘

Refresh Token (Opaque, 256-bit random, 7d):
  存储于 Redis: refresh:{user_id}:{device_id} → {token_hash}
  支持远程吊销和并发会话管理
```

## 4. 性能设计

### 4.1 缓存策略

| 缓存对象 | 存储 | TTL | 命中率目标 |
|:---------|:-----|:----|:-----------|
| JWT 公钥 | 本地缓存 | 1h | >99.9% |
| 用户基本信息 | Redis | 30min | >95% |
| OAuth Client 配置 | Redis | 5min | >98% |
| 速率限制计数器 | Redis (Lua) | 窗口周期 | 100% |
| RBAC 权限树 | Redis + 本地 | 10min | >99% |

### 4.2 连接池配置

| 资源 | 池大小 | 超时 |
|:-----|:-------|:-----|
| PostgreSQL HikariCP | 20-50 per service | 30s |
| Redis Lettuce | 10-30 per service | 5s |
| gRPC Channel | 5-10 per target | 10s |
| HTTP Client (WebClient) | 100 max | 15s |

### 4.3 扩展策略

```
                     ┌──────────────┐
                     │   Load       │
                     │   Balancer   │
                     └──────┬───────┘
                            │
           ┌────────────────┼────────────────┐
           │                │                │
    ┌──────┴──────┐  ┌──────┴──────┐  ┌──────┴──────┐
    │ Auth Pod x3 │  │ User Pod x3 │  │Device Pod x3│
    │ (stateless) │  │ (stateless) │  │ (stateless) │
    └──────┬──────┘  └──────┬──────┘  └──────┬──────┘
           │                │                │
           └────────────────┼────────────────┘
                            │
              ┌─────────────┴─────────────┐
              │    PostgreSQL Cluster     │
              │  Primary + Read Replicas  │
              │  (shard by user_id hash)  │
              └───────────────────────────┘
```

- **水平扩展**：所有服务无状态，通过 K8s HPA 自动扩缩容
- **数据库分片**：按 user_id 一致性哈希分 16 片
- **读写分离**：认证服务读副本，注册/修改走主库

## 5. 高可用设计

| 方面 | 设计 |
|:-----|:-----|
| **多 AZ 部署** | 至少 3 个可用区，每个服务 ≥2 副本 |
| **数据库** | PostgreSQL 主从 + 自动故障转移 (Patroni) |
| **缓存** | Redis Sentinel 哨兵模式，自动主从切换 |
| **消息队列** | RabbitMQ Quorum Queue，镜像队列 |
| **优雅关闭** | Spring Boot Graceful Shutdown + PreStop Hook |
| **健康检查** | Liveness/Readiness Probe，自动重启 |
